# 项目阶段提醒口径统一方案

> 状态：已按用户确认口径进入实施；本文保留原因、统一口径、影响面、逻辑示意和验证清单。  
> 范围：项目当前阶段、下一提醒、紧急/非紧急待处理队列、项目详情弹窗、项目列表、首页生命周期、团队工作完成度与相关文档口径。  
> 核心原则：钉钉表单时间字段是严谨准确的事实来源；系统只负责把这些事实统一解释成阶段、提醒和数据缺口，不反向质疑钉钉字段本身。

## 1. 背景与目标

当前系统已经有多个模块在判断项目处于什么阶段、下一步应该提醒什么：

- 项目详情弹窗展示“当前阶段 / 下一提醒 / 日期与提醒”。
- 项目工作台和下钻列表展示阶段、下一提醒，并按阶段排序。
- 团队工作完成度页展示“紧急项目待处理 Top5 / 非紧急项目待处理 Top5”。
- 首页生命周期模块把项目归入待上会、待复尺、平面方案、施工图、点位设计、采购推进、摆场交付、闭环完成等阶段。
- 团队完成度后端统计平面方案、方案摆场、项目总闭环的完成和进行中状态。

问题在于这些模块目前不是完全复用同一张口径表，而是存在局部重复判断。例如：

- `摆场开始时间` 已在钉钉原始字段中存在，但很多前端/后端别名表不认它。
- `软装完成情况` 漏填时，下一提醒可能卡在“待软装完成情况”，即使下游已经有采购或摆场事实。
- 有的模块把“摆场文件发出时间”当摆场相关证据，有的模块只看“摆场时间/现场摆场时间”。
- 项目详情弹窗、紧急队列、首页生命周期、团队完成度可能对同一个项目给出不同阶段/提醒。

本次目标不是局部修一个字段，而是建立“有且只有一个阶段提醒口径表”，让不同页面复用同一套判定。

## 2. 本次确认的业务口径

### 2.1 钉钉字段是事实，不是建议值

钉钉表单里的时间字段视为准确记录：

- 填了某个节点时间，说明该节点事实已经发生。
- 系统不应因为上游字段漏填而否定下游已发生事实。
- 系统可以提示上游字段存在数据缺口，但不能让数据缺口覆盖真实阶段。

示例：

- `软装完成情况` 空，但 `摆场开始时间` 已填：项目已经进入摆场中；主提醒不能再是“待软装完成情况”。
- `产品清单发出时间` 已填，但 `软装完成情况` 空：项目至少已进入产品清单之后；软装完成情况作为补填项。
- `采购时间` 或 `采购完成情况` 已填，但 `产品清单发出时间` 空：采购事实优先，产品清单作为数据缺口。

### 2.2 摆场开始时间的含义

`摆场开始时间` 是现场摆场开始事实。

口径：

- 只要 `摆场开始时间` 已填写，就表示项目从填入那一刻起进入“摆场中”。
- `摆场时间`、`现场摆场时间` 若存在，也按摆场开始事实处理。
- 系统不应再要求当前日期必须晚于该日期才进入摆场中；因为这个字段不是计划排期，而是事实记录。
- 如极少数出现未来日期，优先仍按钉钉事实处理；可另行作为日期异常或待核对提示，但不应阻断阶段推进。

### 2.3 摆场文件发出时间的含义

`摆场文件发出时间(项目群）` / `摆场文件发出时间（项目群）` 默认表示摆场结束。

口径：

- 摆场文件已经发出，即默认现场摆场已结束。
- 如果项目没有闭环，主提醒应从“等待摆场结束”推进为“项目待闭环”。
- 如果项目已闭环，则不再展示下一提醒。
- 该字段可作为“方案摆场完成量”的完成时间证据。

### 2.4 下游事实可以反推当前阶段

阶段判定不能局限于“当前阶段字段”或某一个完成字段。

很多阶段都可以参考下游事实：

- 有复尺时间，说明上会至少已经完成或已越过上会节点。
- 有平面开始时间，说明复尺已越过。
- 有施工图审核完成时间，说明平面方案和施工图至少已越过。
- 有点位完成时间，说明点位阶段已越过。
- 有软装方案开始时间，说明点位至少已进入或已越过。
- 有产品清单发出/接收时间，说明软装方案已进入下游。
- 有采购时间或采购完成情况，说明已进入采购或采购后。
- 有摆场开始时间，说明已进入摆场中。
- 有摆场文件发出时间，说明摆场结束。

因此系统应采用“最远下游事实优先”的阶段判定方式。

| `lifecycleClosed` | `硬装项目进度=闭环` 或 `软装项目进度=闭环` | 项目闭环 | 退出推进池 |

### 3.2 不再混用的概念

以下概念必须分开：

| 概念 | 解释 | 不能替代 |
| --- | --- | --- |
| 公司阶段 | 项目当前在公司流程里的位置 | 不能直接当项目完成 |
| 下一提醒 | 当前最需要推进的动作 | 不能被普通数据缺口覆盖 |
| 数据缺口 | 上游字段漏填、状态未维护 | 不能否定下游事实 |
| 设计责任闭环 | 设计师或责任线完成交付 | 不能等同项目闭环 |
| 摆场结束 | 摆场文件发出 | 不能被摆场开始替代 |
| 项目闭环 | 公司最终推进结束 | 不能被单个阶段完成替代 |

## 4. 阶段状态机

### 4.1 总体优先级

```text
暂停 / 取消
  > 项目闭环
  > 摆场结束
  > 摆场中
  > 采购中 / 待采购完成
  > 产品清单
  > 软装方案
  > 点位
  > 施工图
  > 平面方案
  > 复尺
  > 上会
  > 未开始
```

说明：

- 越靠上的事实越下游，优先级越高。
- 下游事实存在时，上游未填只形成数据缺口。
- 暂停/取消是业务停止状态，优先级最高。
- 闭环表示项目退出推进，不再展示下一动作提醒。

### 4.2 当前阶段判定

```text
if 当前硬装/软装项目进度为暂停或取消:
  currentStage = 暂停/取消
  primaryReminder = 项目暂停中 / 项目已取消

else if 硬装项目进度=闭环 or 软装项目进度=闭环:
  currentStage = 闭环完成
  primaryReminder = --

else if displayFileSent 已填写:
  currentStage = 摆场结束
  primaryReminder = 项目待闭环

else if displayStart 已填写:
  currentStage = 摆场中
  primaryReminder = 等待摆场结束

else if purchaseStatus 显示完成 or purchaseTime 已填写:
  currentStage = 采购推进
  primaryReminder = 待摆场

else if productListSent 已填写:
  currentStage = 产品清单
  primaryReminder = 待采购完成

else if softDoneStatus 为 准时完成/延期完成 or softDoneTime 已填写:
  currentStage = 软装完成
  primaryReminder = 待产品清单接收

else if softSchemeStart 已填写 or 软装项目进度包含软装方案:
  currentStage = 软装方案
  primaryReminder = 待软装完成

else if pointDone 已填写 or pointStatus 显示完成:
  currentStage = 点位完成
  primaryReminder = 待软装方案

else if constructionReview 已填写:
  currentStage = 施工图审核完成
  primaryReminder = 待点位/软装推进

else if constructionDraft 已填写:
  currentStage = 施工图中
  primaryReminder = 待施工图审核

else if floorPlanFinish 已填写:
  currentStage = 施工图中
  primaryReminder = 待施工图初稿

else if floorPlanStart 已填写:
  currentStage = 平面方案
  primaryReminder = 待平面结束

else if measureDate 已填写:
  currentStage = 平面待启动
  primaryReminder = 待平面开始

else if meetingDate 已填写:
  currentStage = 待复尺
  primaryReminder = 待复尺

else:
  currentStage = 待上会
  primaryReminder = 待上会
```

### 4.3 摆场专项状态

| 条件 | 阶段 | 主提醒 | 数据缺口 |
| --- | --- | --- | --- |
| `displayStart` 空，`displayFileSent` 空，采购未完成 | 采购前 | 按采购或上游节点提醒 | 按缺失字段提示 |
| `displayStart` 空，`displayFileSent` 空，采购已完成 | 待摆场 | 待摆场 | 可提示摆场开始时间待填 |
| `displayStart` 已填，`displayFileSent` 空 | 摆场中 | 等待摆场结束 | 上游漏填只作为数据缺口 |
| `displayStart` 空，`displayFileSent` 已填 | 摆场结束 | 项目待闭环 | 可提示摆场开始时间待补填 |
| `displayStart` 已填，`displayFileSent` 已填 | 摆场结束 | 项目待闭环 | 无摆场主缺口 |
| 项目进度已闭环 | 闭环完成 | -- | 不再展示推进提醒 |

### 4.4 示例：用户截图项目

事实：

```text
项目：北京通州爱琴海店
摆场开始时间：2026-06-07
摆场文件发出时间：空
软装完成情况：空或漏填
系统运行日：2026-06-12
```

判定：

```text
currentStage = 摆场中
primaryReminder = 等待摆场结束
dataGaps = [
  软装完成情况待补填
]
```

页面应显示：

```text
当前阶段：摆场中
下一提醒：等待摆场结束
日期与提醒：
  摆场开始时间：2026-06-07
  摆场文件发出时间：--
  软装完成情况：--
```

不应显示：

```text
下一提醒：待软装完成情况
当前阶段：待摆场
```

## 5. 主提醒与数据缺口的关系

### 5.1 主提醒只保留一个

每个项目在列表、详情和队列中应有一个主提醒。

主提醒用于回答：

> 这个项目当前最应该推进什么？

主提醒优先级：

```text
暂停/取消
  > 系统 Deadline P1/P2
  > 公司阶段下一动作
  > 数据缺口
```

说明：

- 暂停/取消直接覆盖其他提醒。
- 系统 Deadline 严重提醒仍可优先展示，例如平面 Deadline 已延期。
- 公司阶段下一动作表示真实推进动作，例如“等待摆场结束”。
- 数据缺口只有在没有更重要推进动作时才可成为主提醒。

### 5.2 数据缺口可以有多个

数据缺口用于回答：

> 哪些上游字段需要补录或维护？

例如项目已进入摆场中，但上游软装字段漏填：

```text
主提醒：等待摆场结束
数据缺口：
  - 软装完成情况待补填
  - 产品清单接收时间待补填
```

数据缺口不改变阶段。

### 5.3 下游事实对上游缺口的处理

规则：

- 下游事实推进阶段。
- 上游缺口留作补录提醒。
- 不允许上游缺口把阶段拉回去。

示例：

| 下游事实 | 上游缺口 | 阶段 | 主提醒 | 附属提示 |
| --- | --- | --- | --- | --- |
| `摆场开始时间` 已填 | `软装完成情况` 空 | 摆场中 | 等待摆场结束 | 软装完成情况待补填 |
| `采购时间` 已填 | `产品清单发出时间` 空 | 采购推进 | 待采购完成或待摆场 | 产品清单时间待补填 |
| `施工图审核完成时间` 已填 | `施工图初稿完成时间` 空 | 施工图审核完成 | 待点位/软装推进 | 施工图初稿时间待补填 |
| `摆场文件发出时间` 已填 | `摆场开始时间` 空 | 摆场结束 | 项目待闭环 | 摆场开始时间待补填 |

## 6. 推荐技术设计

### 6.1 新增统一口径模块

建议新增一个领域模块：

```text
public/domain/project-stage-reminder-rules.mjs
```

职责：

- 统一字段别名。
- 从项目 rawFields 读取节点事实。
- 生成 `workflowFacts`。
- 根据最远下游事实生成 `currentStage`。
- 生成 `primaryReminder`。
- 生成 `dataGaps`。
- 暴露纯函数供前端页面和后端预计算复用。

不做：

- 不渲染 HTML。
- 不访问 DOM。
- 不发网络请求。
- 不直接修改钉钉字段。
- 不做人工纠错。

### 6.2 统一输出结构

建议统一输出：

```js
{
  currentStage: {
    key: 'displayInProgress',
    label: '摆场中',
    rank: 80,
    evidence: [
      { nodeKey: 'displayStart', field: '摆场开始时间', value: '2026-06-07' }
    ]
  },
  primaryReminder: {
    key: 'waitDisplayEnd',
    type: 'stage_action',
    label: '摆场',
    message: '等待摆场结束',
    severity: 'normal',
    discipline: 'followup',
    evidenceNode: 'displayStart'
  },
  reminders: [
    {
      key: 'waitDisplayEnd',
      type: 'stage_action',
      message: '等待摆场结束'
    },
    {
      key: 'softDoneStatusMissing',
      type: 'data_gap',
      message: '软装完成情况待补填'
    }
  ],
  dataGaps: [
    {
      key: 'softDoneStatusMissing',
      fieldNames: ['软装完成情况'],
      message: '软装完成情况待补填'
    }
  ],
  facts: {
    displayStarted: true,
    displayStartedAt: '2026-06-07',
    displayEnded: false,
    displayEndedAt: '',
    lifecycleClosed: false
  }
}
```

### 6.3 兼容硬装 Deadline

硬装 Deadline 现有 `primaryReminder` 不能丢。

合并规则：

```text
1. 先生成 workflowReminderResult。
2. 再读取 hardDeadline / project.primaryReminder。
3. 如果 hardDeadline 是 P1/P2 严重 open 提醒，则它可以成为列表主提醒。
4. workflow 的阶段动作仍保留在 reminders 中。
5. 如果 hardDeadline 是 manual_review / missing_field 且只是规则基础缺失，则不抢占公司阶段下一动作。
```

示意：

```text
项目同时存在：
  - 系统平面 Deadline 已延期
  - 摆场中，等待摆场结束

列表主提醒：
  系统平面 Deadline 已延期

详情提醒堆栈：
  1. 系统平面 Deadline 已延期
  2. 等待摆场结束
```

### 6.4 catalog/read-model 需要带轻量结果

项目 catalog summary 可能不带完整 rawFields。

如果列表、首页、下钻弹窗要一致，预计算读模型必须写入轻量字段：

```js
{
  stageReminder: {
    currentStage: {...},
    primaryReminder: {...},
    dataGapCount: 2
  },
  workflowFacts: {
    displayStartedAt: '2026-06-07',
    displayEndedAt: ''
  }
}
```

原则：

- 不把完整 rawFields 塞回 summary。
- 只写页面和排序必要的轻量结果。
- full detail 仍可返回完整 rawFields，用于弹窗详情。

如果项目已闭环：

### 8.2 紧急/非紧急待处理队列

队列分组仍只看人工优先级：

```text
紧急：项目状态/紧急程度含“紧急”，且不含“不紧急/非紧急/一般”
非紧急：未标记紧急或标记一般/非紧急
```

`actionStage` 改用统一口径：

```text
摆场开始已填、摆场文件未发出：
  actionStage = 等待摆场结束

摆场文件已发出、未闭环：
  actionStage = 项目待闭环

采购已完成、摆场未开始：
  actionStage = 待摆场
```

### 8.3 首页生命周期

分类应使用统一 `currentStage.key`：

| currentStage | 首页阶段 |
| --- | --- |
| `notStarted` | 待上会 |
| `meeting` | 待复尺 |
| `floorPlan` | 平面方案 |
| `construction` | 施工图 |
| `point` | 点位设计 |
| `softScheme` | 方案设计 |
| `purchase` | 采购推进 |
| `displayInProgress` | 摆场交付 |
| `displayFinished` | 摆场交付 |
| `closed` | 闭环完成 |
| `paused/canceled` | 暂停/取消 |

### 8.4 团队完成度

方案摆场相关状态：

```text
display.inProgress:
  displayStart 已填，displayFileSent 未填，项目未闭环
  或软装/采购阶段已推进但摆场未结束

display.completed:
  displayFileSent 已填

display.none:
  未到摆场相关阶段
```

注意：

- `摆场开始时间` 只能让项目进入摆场进行中。
- `摆场文件发出时间` 才默认表示摆场结束。
- 项目闭环另算 `lifecycle.completed`，不能反推摆场完成时间。

## 11. 验收标准

### 11.1 业务验收

同一个项目在以下页面口径一致：

- 首页生命周期。
- 项目工作台。
- 项目详情弹窗。
- KPI 下钻弹窗。
- 团队工作完成度。
- 紧急/非紧急待处理队列。

### 11.2 关键断言

必须成立：

```text
摆场开始时间已填 + 摆场文件未发出
  => 摆场中 / 等待摆场结束

摆场文件已发出
  => 摆场结束 / 项目待闭环

软装完成情况漏填 + 下游摆场事实存在
  => 不阻断摆场阶段，只作为数据缺口

暂停/取消
  => 覆盖所有推进提醒

闭环
  => 不再展示下一提醒
```

## 12. 风险与边界

### 12.1 风险

- 这是规则口径变更，不是单点 UI 修复。
- 因为多个页面会复用统一结果，一处判断错误会全局扩散。
- 旧测试中有些断言锁定了旧口径，例如“软装完成情况优先于下游采购/摆场提醒”，需要改掉。
- project catalog summary 若不带新轻量字段，首屏和详情可能再次分裂。

### 12.2 边界

- 本次不默认修改钉钉表单字段含义。
- 本次不反向写钉钉。
- 本次不把普通数据缺口当成阶段事实。
- 本次不把项目闭环与摆场结束混为一谈。
- 本次不把人工紧急状态用于阶段判定；紧急仍只影响队列分组和优先处理。

## 13. 最终口径摘要

一句话版本：

> 系统以钉钉时间字段为事实，用最远下游事实判断项目当前阶段；主提醒只表达当前最该推进的动作，字段漏填作为数据缺口另行提示。填了摆场开始时间即进入摆场中，提醒等待摆场结束；摆场文件发出即默认摆场结束，下一步提醒项目待闭环。
