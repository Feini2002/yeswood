# 小组页团队工作完成情况口径切换延时修复方案

> 状态：方案稿，待确认后实施  
> 目标页面：小组页 / 团队工作完成情况模块  
> 目标交互：点击「直营 / 加盟」后，Tab 与内容尽量同步切换；命中本地缓存时不等待网络；未命中缓存时保留旧内容占位，不清空页面。  
> 本次范围：前端加载策略、缓存命中路径、请求防回写、回归测试。暂不改钉钉字段语义、后端统计口径或 read model 生成逻辑。

## 1. 背景

当前小组页的「团队工作完成情况」模块支持按负责人、直营 / 加盟口径和年份查看团队、小组、成员的完成情况。用户反馈的卡点是：点击「直营 / 加盟」后，按钮状态先变，但数字、分组、趋势图等内容要等网络请求回来后才变化，体感像卡了一下。

截图里的复盘基本准确：当前入口是 UI 先行、数据后置。

```text
点击直营 / 加盟
  -> syncTeamCompletionControls() 先切 Tab
  -> navigateTeamWorkCompletion() 更新 hash
  -> loadTeamDashboardScope()
  -> loadTeamDashboardSessionBundle()
  -> 等 /api/dashboard-session 返回
  -> renderTeamDashboard()
  -> renderTeamWorkCompletionDashboard()
  -> renderOwnerReviewDashboard()
```

这条链路的问题不是口径错，而是点击入口没有优先使用内存缓存。即使之前已经访问过目标口径，当前按钮切换仍然会先请求 `/api/dashboard-session`，导致「按钮先变、内容慢半拍」。

## 2. 当前代码事实

### 2.1 点击入口

`public/pages/teams.mjs` 中的 `handleTeamWorkCompletionContextClick(event)` 做了三件事：

1. 读取按钮上的 `data-team-completion-context`。
2. 调用 `syncTeamCompletionControls(state.teamWorkCompletion, dashboardContext)`，让目标 Tab 立即高亮。
3. 更新 hash 后调用 `loadTeamDashboardScope(owner, dashboardContext, year)`。

核心问题在第 3 步：它没有先查目标 `owner + dashboardContext + year` 是否已经有本地缓存。

### 2.2 当前口径切换主链路

`loadTeamDashboardScope(owner, dashboardContext, year)` 当前优先调用 `loadTeamDashboardSessionBundle(owner, dashboardContext, year)`。这个函数直接请求 `/api/dashboard-session`，并在返回后同时写入三块状态：

- 负责人项目盘面：`state.teamMetrics`
- 团队工作完成情况：`state.teamWorkCompletion`
- 负责人负载复盘：`state.ownerReview`

所以口径按钮切换被绑定到一个「整页 session bundle」请求上，而不是只围绕当前点击的团队完成模块做本地快切。

### 2.3 已有但没有被入口优先使用的能力

`loadTeamWorkCompletion(owner, dashboardContext, year, options)` 已经具备更适合按钮切换的策略：

- `cachedTeamWorkCompletion(owner, dashboardContext, year)` 命中时立即回填并渲染。
- 未缓存但同 owner 时，可以保留旧 review 作为 stale 占位。
- 未缓存且换 owner 时，会阻止旧 owner 数据继续显示。
- 使用 `runtimeStore.teamWorkCompletionRequestId` 和 request map 防止旧请求回写。

`loadTeamPageModules({ forceRefresh = false })` 也有 session bundle 的 cache-first 判断：

```text
hasLoadedTeamSessionBundle(owner, dashboardContext, year)
  = teamMetricsByOwner[owner]
    + cachedTeamWorkCompletion(owner, dashboardContext, year)
    + cachedOwnerReview(owner, dashboardContext)
```

但这个快路径只在页面模块加载中使用，点击直营 / 加盟时没有走到。

### 2.4 现有测试已经锁住的行为

`tests/publicAppBehavior.test.mjs` 已经覆盖了几类关键行为：

- `team work completion keeps current dashboard visible while switching uncached scope`
  - 未缓存口径切换时，保留旧内容占位，不显示大面积 loading。
- `team dashboard scope switch keeps session preparing from falling back to module fanout`
  - `/api/dashboard-session` 处于 preparing 时，不回退到三接口 fanout。
- `team dashboard session preparing shows target owner loading instead of stale owner data`
  - 换 owner 时不能继续显示旧 owner 数据。
- `initial teams route loads dashboard session and preloads current detail without page fanout`
  - 首屏 teams 仍应走 dashboard-session，避免 showPage + loadDashboard 双加载。

因此修复不能简单把旧内容清掉、不能绕过 session preparing 保护，也不能破坏首屏加载边界。

## 3. 根因判断

延时来自加载策略的预期副作用：

| 现象 | 根因 |
| --- | --- |
| Tab 先变、内容后变 | 点击入口先执行 UI pending context，再等待 session bundle |
| 已访问过的口径仍要等网络 | `handleTeamWorkCompletionContextClick -> loadTeamDashboardScope` 没有目标口径 cache-first |
| 等待时像卡住但不是 loading | 同 owner 不清空旧内容，这是已有防闪烁策略 |
| 返回后重绘明显 | session bundle 同时重绘负责人盘面、完成情况、负责人负载 |
| 加盟口径 0 会显得跳 | 真实数据从 146 项变 0 项，视觉差异大，且结果等网络回来才出现 |

结论：要优化的是「口径切换入口的加载策略」，不是直营 / 加盟业务口径，也不是直接删除旧数据占位逻辑。

## 4. 修复目标

### 4.1 用户体验目标

- 命中目标口径缓存时，点击后同一帧或下一个 microtask 内完成内容切换。
- 未命中缓存时，继续保留当前可见内容，避免大面积 loading。
- Tab 与内容不长期不一致；若内容仍是旧口径，应给出轻量 refresh / stale 状态，而不是空白。
- 快速来回切换时，旧请求不能覆盖当前口径。

### 4.2 架构目标

- `showPage`、`hashchange`、首屏 `loadDashboard` 的边界不变。
- 口径按钮切换使用 cache-first，但 session bundle 仍可作为后台校验和补齐来源。
- 不引入新的后端接口。
- 不扩大 `/api/projects` 或三接口 fanout。
- 不破坏 `/api/dashboard-session` preparing 时的保护。

## 5. 推荐方案：本地快切 + session 后台校验

### 5.1 核心思路

把口径切换拆成两段：

```text
点击目标口径
  -> 先同步 Tab 和 hash
  -> 尝试应用目标口径内存缓存
      - 命中：立即渲染目标口径团队完成模块
      - 未命中：保留旧内容占位
  -> 后台请求 dashboard-session
  -> 响应仍匹配当前 hash 时，补齐三块模块并重绘
  -> 响应已过期时丢弃
```

这保留了 session bundle 的整页一致性，同时让用户感知最快的「团队工作完成情况」模块先切换。

### 5.2 新增一个小的本地应用函数

建议在 `public/pages/teams.mjs` 新增内部 helper，例如：

```javascript
function applyCachedTeamDashboardScope(owner, dashboardContext, year) {
  const normalizedContext = resolveTeamPageDashboardContext(dashboardContext);
  const normalizedYear = Number(year) || new Date().getFullYear();
  const cachedCompletion = cachedTeamWorkCompletion(owner, normalizedContext, normalizedYear);
  const cachedReview = cachedOwnerReview(owner, normalizedContext);
  const canUseMetrics =
    state.teamMetricsBatchKey === teamMetricsContextKey(normalizedContext) &&
    state.teamMetricsByOwner?.[owner];

  let applied = false;

  if (cachedCompletion) {
    state.teamWorkCompletion = cachedCompletion;
    state.teamWorkCompletionYear = cachedCompletion.year || normalizedYear;
    state.teamWorkCompletionLoading = false;
    state.teamWorkCompletionError = '';
    state.teamWorkCompletionRefreshStatus = 'refreshing';
    state.teamWorkCompletionRefreshError = '';
    state.teamWorkCompletionSwitchTarget = '';
    applied = true;
  }

  if (cachedReview) {
    state.ownerReview = cachedReview;
    state.ownerReviewLoading = false;
    state.ownerReviewError = '';
    state.ownerReviewRefreshStatus = 'refreshing';
    state.ownerReviewRefreshError = '';
    applied = true;
  }

  if (canUseMetrics) {
    state.teamMetrics = state.teamMetricsByOwner[owner];
    state.teamMetricsLoading = false;
    state.teamMetricsError = '';
    applied = true;
  }

  return applied;
}
```

注意：这里不能为了查负责人盘面缓存调用 `ensureTeamMetricsCacheContext(normalizedContext)`。该函数在 context 不一致时会清空 `state.teamMetricsByOwner`，会把当前口径缓存冲掉，反而造成闪烁。

### 5.3 给 scope 切换加 requestId

当前 `loadTeamWorkCompletion` 自己有 requestId，但 `loadTeamDashboardScope -> loadTeamDashboardSessionBundle` 这层没有独立防回写。建议在 `runtimeStore` 增加：

```javascript
teamDashboardScopeRequestId: 0
```

`loadTeamDashboardScope` 每次非后台 scope 切换时递增 requestId：

```javascript
const requestId = ++runtimeStore.teamDashboardScopeRequestId;
```

session bundle 返回后先确认：

```javascript
if (requestId !== runtimeStore.teamDashboardScopeRequestId) {
  return null;
}

if (!teamDashboardScopeStillCurrent(owner, resolvedDashboardContext, year)) {
  return null;
}
```

`teamDashboardScopeStillCurrent` 用当前 hash / resolver 对比 `owner + dashboardContext + year`。这样快速点击 direct -> franchise -> direct 时，franchise 的慢响应不会覆盖 direct。

### 5.4 调整 `loadTeamDashboardScope`

推荐改成：

```text
loadTeamDashboardScope(owner, dashboardContext, year)
  1. 参数规范化
  2. 尝试 applyCachedTeamDashboardScope()
  3. 如果当前页是 teams 且命中缓存，立即 render 三块中已命中的部分
  4. 再请求 loadTeamDashboardSessionBundle()
  5. preparing：保持现有逻辑，不 fallback fanout
  6. 成功：仍匹配当前 scope 时统一 render 三块
  7. session 失败：如果已有本地缓存，保留 stale 状态；如果完全没缓存，再走现有 fallback 或错误逻辑
```

关键边界：

- 命中 completion 缓存时，团队完成模块应马上显示目标口径。
- 负责人盘面 metrics 如果当前 cache context 不匹配，可以不秒切，等 session 回来。
- owner review 命中缓存时可一起秒切；未命中则保留旧模块或轻量 refresh 状态。
- session preparing 不能触发 `loadTeamMetrics + loadTeamWorkCompletion + loadOwnerResponsibilityReview` 三接口 fanout。

### 5.5 点击入口保持薄

`handleTeamWorkCompletionContextClick` 仍只负责：

- 阻止默认事件；
- 解析目标 context；
- 同步 Tab pending 状态；
- 更新 hash；
- 调用 `loadTeamDashboardScope`。

不要在点击函数里堆缓存细节，否则后续 year change、owner change、页面切换会出现多套策略。

## 6. 备选方案

### 方案 A：只让 `loadTeamDashboardScope` 复用 `hasLoadedTeamSessionBundle`

做法：把 `hasLoadedTeamSessionBundle` 或等价 helper 从 `dashboard-loader.mjs` 抽出或导出，`loadTeamDashboardScope` 开头先判断完整三块缓存，命中则直接渲染。

优点：

- 改动小。
- 和 `loadTeamPageModules` 的整包缓存语义一致。

缺点：

- 必须三块都命中才秒切；如果 completion 已有缓存但 owner review 或 metrics 缺失，用户仍要等。
- 对当前痛点不够精准。

### 方案 B：点击后直接走 `loadTeamWorkCompletion`

做法：`handleTeamWorkCompletionContextClick` 不再先走 session bundle，而是先调用 `loadTeamWorkCompletion(owner, context, year)`，再后台刷新其它模块。

优点：

- 团队完成模块响应最快。
- 复用已有 completion 缓存和 requestId。

缺点：

- 点击入口会变重。
- 容易让三块模块的加载编排分裂。
- 需要额外处理 session preparing 和 dashboard-session 首屏策略，风险更高。

### 方案 C：推荐方案，本地快切 + session 后台校验

优点：

- completion 缓存命中即可秒切，不要求三块全部命中。
- 不破坏 session bundle 的整页一致性。
- 可以在 `loadTeamDashboardScope` 一处收敛 owner/context/year 切换策略。
- 保留未缓存时的旧内容占位策略。

缺点：

- 需要新增 scope requestId 和 current scope 校验。
- 需要补充更细的测试，防止旧响应回写。

推荐采用方案 C。

## 7. 可选二期：预热当前负责人的另一个口径

如果要让「第一次点击加盟」也尽量秒切，可以在当前 owner 的 direct 或 franchise summary 已加载后，低优先级预热另一个 context：

```text
当前 owner + direct 加载完成
  -> 延迟 2-3 秒
  -> 如果仍在 teams 页且空闲
  -> background 请求 owner + franchise + year 的 team-work-completion summary
  -> 写入 cachedTeamWorkCompletion
```

注意事项：

- 这只能作为二期优化，不应和本次主修复绑死。
- 预热必须可取消，不能影响首屏。
- 不应把项目 catalog 或三接口重接口放入同一 `Promise.all`。
- 预热失败只记录 warning，不影响页面。

## 8. 测试方案

修改加载边界时，按项目规则至少更新并运行：

- `tests/publicAppBehavior.test.mjs`
- `tests/frontendSplitPolicy.test.mjs`（如果职责边界或守则文本变更）
- `tests/frontendLoadPerformancePolicy.test.mjs`（如果触及 dashboard-loader 或项目加载边界）

建议新增 / 调整以下用例：

### 8.1 已缓存目标口径时不等网络

场景：

1. 当前 hash 是 `#teams?owner=Owner%20A&dashboardContext=direct&year=2026`。
2. `state.teamWorkCompletionByKey` 已有 `Owner A + franchise + 2026`。
3. `/api/dashboard-session?context=franchise` 人为延迟。
4. 点击 franchise。

断言：

- franchise Tab 立即 active。
- 在 session promise resolve 前，`state.teamWorkCompletion.dashboardContext === 'franchise'`。
- 页面 HTML 已出现 franchise payload 的小组 / 数字。
- pending 阶段不出现大面积 loading。
- session resolve 后仍是 franchise。

### 8.2 未缓存目标口径时保留旧内容

保留现有测试语义：

- direct 内容仍作为占位可见；
- franchise Tab active；
- 不显示大面积 loading；
- session 返回后再切到 franchise 内容。

### 8.3 preparing 不 fanout

沿用并强化：

- `/api/dashboard-session` 返回 `{ status: 'preparing' }`。
- 不请求 `/api/team-metrics-batch`、`/api/team-work-completion`、`/api/team-responsibility-review`。
- 已有同 owner 内容可保留，并标记 preparing。

### 8.4 快速切换旧响应不回写

场景：

1. direct -> franchise 发出慢请求 A。
2. franchise -> direct 发出快请求 B。
3. B 先返回，A 后返回。

断言：

- 最终 hash 是 direct。
- `state.teamWorkCompletion.dashboardContext === 'direct'`。
- A 返回后不覆盖 direct。
- `renderTeamWorkCompletionDashboard` 不把过期 franchise 画回页面。

### 8.5 metrics context 不被误清空

场景：

- 当前 `state.teamMetricsBatchKey === 'direct'`，`state.teamMetricsByOwner` 有 direct 缓存。
- 点击 franchise，completion 命中缓存，但 metrics context 不匹配。

断言：

- 不调用会清空 direct metrics 的路径。
- 当前负责人盘面不会被错误置空。
- session 返回前不把 direct metrics 当 franchise metrics 展示。

## 10. 风险与控制

| 风险 | 控制 |
| --- | --- |
| metrics 缓存按 context 分桶，误用会显示错口径 | 只有 `state.teamMetricsBatchKey` 匹配时才复用 metrics |
| session 慢响应覆盖当前口径 | 新增 scope requestId + 当前 hash 校验 |
| preparing 时回退三接口，破坏 read model 策略 | 保留现有 preparing 直接返回逻辑 |
| completion 命中但 review 未命中造成三块不同步 | completion 先秒切，review / metrics 标记 refreshing 或等待 session；不假装三块都已切完 |
| 预热增加首屏压力 | 预热放二期，延迟、可取消、低优先级 |

## 11. 建议实施步骤

1. 在 `runtimeStore` 增加 `teamDashboardScopeRequestId`。
2. 在 `teams.mjs` 增加 `teamDashboardScopeStillCurrent`。
3. 在 `teams.mjs` 增加 `applyCachedTeamDashboardScope`，只做目标缓存读取和状态回填。
4. 调整 `loadTeamDashboardScope`：
   - 请求 session 前先应用目标缓存；
   - 命中缓存立即渲染；
   - session 返回后校验 requestId 和当前 scope；
   - preparing 和 fallback 行为保持现有边界。
5. 为目标缓存快切、未缓存占位、preparing 不 fanout、旧响应不回写补测试。
6. 运行相关测试。
7. 由用户当前 Chrome 页面完成视觉 / 交互验收；Codex 不额外新开浏览器。

## 12. 不建议做的事

- 不建议把未缓存切换改成整块 loading。当前旧内容占位是合理的防闪烁策略。
- 不建议在 `hashchange` 入口直接加载数据。项目规则要求同页 hash/query 变化不触发页面级重载。
- 不建议让点击入口直接并发三接口。这样会回退到旧性能问题。
- 不建议为了加盟为 0 单独改业务口径。0 是真实数据差异，应通过空态文案解释，而不是改统计逻辑。
- 不建议用新后端接口解决本问题。现有前端缓存和 session bundle 已足够。
