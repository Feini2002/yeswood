# 看板性能架构与防复发治理

本文档汇总中台看板在 **Node 延迟**、**首屏/切换卡顿** 方面的架构设计与防复发约定，供新增页面、接口与数据模块时对照接入。可执行细则以 `AGENTS.md` 中「路由与数据加载解耦」「看板加载性能」两节为准；本文侧重 **体系说明、扩展点与自检清单**。

---

## 1. 目标与范围

| 目标 | 说明 |
| --- | --- |
| 降低 Node 主线程阻塞 | 重计算离线、压缩异步、同步 I/O 最小化 |
| 降低首屏与切换等待 | 聚合接口、预计算读盘、前端 cache-first |
| 防止功能迭代回退性能 | 固定扩展点 + 策略测试，替代周期性全盘 review |

**范围**：`public/` 前端加载链、`src/backend/` API 与预计算、`data/precomputed/` 与 `data/read-model/`、相关测试与启动脚本。

**不在本文范围**：钉钉源数据逻辑、业务规则口径（见 `docs/rules/`）。

## 2. 架构总览

### 2.1 前端三层

```
hashchange / 导航
       ↓
router.mjs          ← 只切页面壳、hash、高亮；不在此拉重数据
       ↓
dashboard-loader.mjs ← 首屏/刷新/soft·hard refresh 唯一编排
       ↓
pages/*.mjs + domain/*.mjs ← 模块级缓存、去重、防回写
```

| 层 | 职责 | 权威文件 |
| --- | --- | --- |
| 路由壳 | 切 tab、应用 hash；`pageChanged` 时才触发页面级 loader | `public/lib/router.mjs` |
| 加载编排 | `loadDashboard` / `loadDashboardSession` / `softRefresh` / `hardRefresh` | `public/lib/dashboard-loader.mjs` |
| 领域缓存 | 项目目录、下钻、详情 | `public/domain/project-catalog.mjs` |
| 运行时去重 | requestId、generation、in-flight Map | `public/lib/runtime-flags.mjs` |

### 2.2 后端四层读路径

对重接口，请求处理顺序固定为：

```
read-model/current（最快）
    ↓ miss
data/precomputed/{snapshotHash}/（预计算文件）
    ↓ miss
进程内 Map 缓存（按 snapshot 签名键控）
    ↓ miss
现场 build*() 计算
```

| 组件 | 职责 | 权威文件 |
| --- | --- | --- |
| 响应出口 | JSON 序列化、>1KB 异步 gzip、性能日志 | `src/backend/server.mjs` → `sendJson` |
| Snapshot | 内存缓存 + in-flight 去重；变更后调度预计算 | `src/backend/syncService.mjs` |
| 预计算 | Worker 线程算完落盘 | `precomputeWorker.mjs`、`precomputeTeamDashboards.mjs` |
| Read Model | 原子发布到 `read-model/current/` | `readModelRepository.mjs` |
| 同步限流 | `reserveSyncGate` 防止并发/频繁钉钉同步拖慢服务 | `syncGate.mjs` |
| 存储 | SQLite WAL；snapshot 临时文件 + rename | `database.mjs`、`storage.mjs` |

### 2.3 预计算 Feature 清单

以下 feature 在 snapshot 变更后由 Worker 预生成，并纳入**双重完整性校验**：

| 常量 | 位置 | 作用 |
| --- | --- | --- |
| `COMPLETE_PRECOMPUTE_FEATURES` | `precomputeTeamDashboards.mjs` | 判定预计算 manifest 是否完整，缺失则重新全量算 |
| `REQUIRED_READ_MODEL_FEATURES` | `readModelRepository.mjs` | 判定 read model 目录是否完整，缺失则 API 退化为 build*() 现场算 |

两个常量包含的 feature 集合一致（均为下列 8 个），新增 feature 时**两处必须同步更新**（见 §5 检查点 6）：

| Feature 常量名 | 用途 |
| --- | --- |
| `team-work-completion` | 团队完成度（主结果） |
| `team-work-completion-summary` | 团队完成度摘要 |
| `team-work-completion-detail` | 团队完成度明细 |
| `team-responsibility-review` | 负责人复盘 |
| `team-metrics` | 小组指标批量 |
| `dashboard-session` | `/api/dashboard-session` 首屏聚合包 |
| `project-catalog-summary` | 项目目录摘要 |
| `profile-dashboard` | 个人盘（direct / franchise / department） |

目录约定：

- 预计算：`data/precomputed/{snapshotHash}/`
- Read model：`data/read-model/current/`
- 启动预热：`GET /api/dashboard-warmup`（`scripts/launch-dev-dashboard.ps1` 等）

## 3. 已落地的性能措施

### 3.1 后端

1. **异步 gzip**：`gzipAsync`，禁止 `gzipSync` 阻塞事件循环（`AGENTS.md` + `frontendLoadPerformancePolicy.test.mjs` 硬性约束）。
2. **Worker 预计算**：`worker_threads` 执行 `precomputeTeamDashboards`，主进程继续接 API；同一 `snapshotHash` 去重调度。
3. **多层内存缓存**（`clearSnapshotCache` 时一并清空）：
   - `snapshotCache` / `snapshotCachePromise`
   - `metricsResponseCache`
   - `teamMetricsCache`、`teamWorkCompletionCache`、`teamResponsibilityReviewCache`
   - `projectsIdsCache`
4. **响应瘦身**：`/api/projects` 默认 `view=summary`；下钻专用 `fields=ids`；单项详情 `view=full` 懒加载。
5. **API 性能观测**：`logApiPerformance` 记录 `readModelHit`、`precomputedFileHit`、`gzipMs`、`payloadKb` 等。
6. **SQLite WAL** 模式、预计算结果原子 rename 落盘、`reserveSyncGate`（`syncGate.mjs`）防止并发同步拖慢主 API。

### 3.2 前端

1. **聚合首屏**：`loadDashboardSession` → `/api/dashboard-session`，一次拿 snapshot、metrics、profile、catalog、team 包。
2. **首屏减负**：`loadCoreDashboard` 仅 **一次** `/api/metrics`，不默认拉全量 `/api/projects`。
3. **softRefresh**：明细/概览筛选、同页 hash 变化 → 本地过滤，禁止误触 `loadDashboard({ forceRefresh: true })`。
4. **下钻路径**：`resolveDrillProjects` + `fields=ids` + 本地 catalog 拼装，禁止全量 projects 列表。
5. **小组页优先级**：`loadTeamPageModules` 先完成 metrics / completion / responsibility，catalog **后台**预载，禁止与重接口同一 `Promise.all` 抢首屏。
6. **切换体验**：cache-first、保留上一份完成度占位、`teamMetricsCacheGeneration` / `AbortController` 防过期响应回写。
7. **刷新边界**：仅用户显式刷新、同步、分析小组、自动更新等传递 `forceRefresh: true`。

### 3.3 治理与测试（替代全盘 review）

| 文档/测试 | 作用 |
| --- | --- |
| `AGENTS.md` § 路由与数据加载解耦 | 人读规则：showPage / loadDashboard / hashchange 边界 |
| `AGENTS.md` § 看板加载性能 | 人读规则：catalog、softRefresh、预计算、gzip |
| `tests/frontendLoadPerformancePolicy.test.mjs` | 静态策略：无双 metrics、无 gzipSync、warmup、catalog 顺序等 |
| `tests/frontendSplitPolicy.test.mjs` | 路由与 loader 职责、forceRefresh、缓存字段存在性 |
| `tests/publicAppBehavior.test.mjs` | 行为回归：首屏无 fanout、hash 不重载、stale 不回写等 |
| `tests/precomputeTeamDashboards.test.mjs` | 预计算产物与 live 计算一致性 |

修改 `router.mjs`、`dashboard-loader.mjs`、`app.js`、`pages/teams.mjs`、`pages/profile-shared.mjs`、`server.mjs`、`project-catalog.mjs` 等边界文件时，须同步跑上述相关测试。

## 4. 体系化程度说明

### 4.1 已成体系（可当架构用）

- 前后端均有 **固定分层** 与 **统一读路径**（预计算 → 缓存 → 现场算）。
- 前端 **唯一编排入口**（`dashboard-loader`）与 **项目数据单一出口**（`project-catalog`）。
- 后端 **统一响应管道**（`sendJson` + 性能日志）。
- **规则 + 测试** 构成防复发闭环，无需依赖周期性人肉全盘 review。

### 4.2 尚未框架化（扩展时需手动接）

1. **页面注册**：新页面在 `showPage` 中仍为按 `pageId` 的 `if` 分支，无统一 `PageLoaderRegistry`。
2. **预计算注册**：新重接口须**手动注册** `COMPLETE_PRECOMPUTE_FEATURES` **与** `REQUIRED_READ_MODEL_FEATURES` 两处常量。
3. **测试覆盖**：测试为**静态策略 + 行为用例**，不会在运行时拦截错误写法。
4. **知识覆盖**：`AGENTS.md` 仅覆盖**已知页面类型**，新类型不会自动受保护。

**结论**：不是「加页面就自动快」，而是「加页面必须走固定管道，否则极易回退」。

## 5. 新功能接入：七个检查点

新增页面、Tab、重接口或下钻时，按下列入口接入，不必全盘 review。

### 检查点 1 — 路由：`public/lib/router.mjs`

- `hashchange` **只**调用 `showPage(currentPageId())`。
- 首屏进入新页面：在 `showPage` 内增加 `!skipPageDataLoad && pageChanged && pageId === '...'` 分支，通过 `routerHooks` 调 loader。
- 同页 query/hash 变化：**不得**触发页面级 reload；交给 `softRefresh`。

**反模式**：在 `app.js` 的 `hashchange` 或筛选 `change` 里直接 `fetchJson` / `loadDashboard`。

### 检查点 2 — 编排：`public/lib/dashboard-loader.mjs`

| 场景 | 接法 |
| --- | --- |
| 首屏必需数据 | `loadDashboard` / `loadDashboardSession`，或 `showPage` 的 pageChanged 分支 |
| owner / context / year 切换 | cache-first + 保留旧 UI，后台拉新 |
| 筛选 / 搜索 / 同页 hash | `softRefresh` 本地过滤 |
| 用户点刷新 / 同步 / 自动更新 | 唯一允许 `forceRefresh: true` |

首屏初始化：`showPage(currentPageId(), { skipPageDataLoad: true })` + `loadDashboard` 单独编排，避免双加载。

### 检查点 3 — 页面模块：`public/pages/*.mjs`

以 `teams.mjs` 为范本，重页面应具备：

- 缓存键（如 `owner + dashboardContext + year`）
- in-flight 去重（`runtimeStore.*Promises`）
- 防回写（`teamMetricsCacheGeneration` / `requestId` / `AbortController`）
- 有旧数据时不整块 loading
- 默认 `forceRefresh: false`

新模块状态优先扩 `public/lib/runtime-flags.mjs`，避免在 `app.js` 新增全局单例。

### 检查点 4 — 项目数据：`public/domain/project-catalog.mjs`

- 列表 → `view=summary`
- 下钻 → `resolveDrillProjects` + `fields=ids`
- 单条详情 → `fetchProjectDetail` + `view=full`
- snapshot 签名变化 → `invalidateProjectCaches`

**反模式**：`fetchJson('/api/projects' + toQuery(filters))` 拉全量列表。

### 检查点 5 — 后端 API：`src/backend/server.mjs`

新重接口遵循：

```
readPrecomputed*() → 内存 Map → build*()
```

参考：`resolveTeamWorkCompletionReview`、`resolveTeamMetricsBatch`。

- 缓存键含 snapshot hash
- 支持 `forceRefresh` / `skipPrecomputed`
- 响应经 `sendJson`（异步 gzip）
- 需要时打 `logApiPerformance`

### 检查点 6 — 预计算（仅慢且可离线算的接口）

新增 feature 须同时更新以下文件（详见 §2.3 双重校验体系）：

1. 在 `precomputeTeamDashboards.mjs`：
   - 定义 `*_PRECOMPUTE_FEATURE` 常量（如 `TEAM_WORK_COMPLETION_PRECOMPUTE_FEATURE = 'team-work-completion'`）
   - 写入 `COMPLETE_PRECOMPUTE_FEATURES` 数组
   - 实现对应的 `write*()` / `compute*()` 逻辑
2. 在 `readModelRepository.mjs`：
   - 定义 `*_READ_MODEL_FEATURE` 常量（如 `TEAM_WORK_COMPLETION_READ_MODEL_FEATURE = 'team-work-completion'`）
   - 写入 `REQUIRED_READ_MODEL_FEATURES` 数组
   - 实现对应的 `write*()` / `read*()` 逻辑
3. 若需首屏聚合，写入 `dashboard-session/core.json`
4. 补充 `tests/precomputeTeamDashboards.test.mjs` 或 API 测试

> **重要**：两套常量必须保持相同 feature 集合，漏更任一数组会导致预计算/read model 完整性检查失效。

轻量接口：进程内缓存即可，不必预计算。

### 检查点 7 — 测试与规则

至少新增 **1 条** `publicAppBehavior.test.mjs` 场景（如：进入不重拉、切换保留旧数据、stale 不回写）。若改动加载边界，更新 `AGENTS.md` 对应小节及 policy 测试中的 assert。

## 6. PR / 功能自检清单

```text
□ 数据加载是否只经过 dashboard-loader / 页面 loader，而不是 hashchange？
□ 同页参数变化是否 softRefresh，而非 loadDashboard(forceRefresh)？
□ 是否有 cache-first + 旧数据占位？
□ 是否有 in-flight 去重 + 过期响应丢弃？
□ 项目列表是否走 project-catalog（summary / ids / full）？
□ 后端是否 precomputed → memory → build，且走 sendJson？
□ 若很重，是否注册 precompute + read model feature？
□ 是否补了 publicAppBehavior / policy 测试？
□ 修改边界文件后是否跑了 AGENTS.md 要求的测试？
```

## 8. 已知残留风险

（来源：`docs/archive/2026-06-review-correctness-report.md` 等，后续优化可跟踪于此节。）

| 项 | 说明 | 可能影响 |
| --- | --- | --- |
| `snapshotCacheKey` 使用 `fs.statSync` | 每次 `getSnapshot` 可能同步 stat | 高负载或网络盘上事件循环微阻塞 |
| 部分路由每次请求开/关 SQLite | 如 team-metrics 相关路径 | 连接创建开销 |
| read model 读盘使用 `readFileSync` | `readModelRepository.readJsonFile` | 单次快，仍为同步 I/O |

这些 **不否定** 当前体系，但新增功能时不应再扩大同步 I/O 与重复建连模式。

## 9. 可选的下一步工程化

当前未实现；若页面与接口继续快速增长，可考虑：

1. **`PageLoader` 注册表**：`{ pageId, onEnter, onHashChange }`，减少 `showPage` 内 if 链。
2. **`createCachedLoader({ key, fetch, staleWhileRevalidate })`**：统一 teams/profile 缓存模式。
3. **后端 `createApiHandler({ precomputeFeature, build, cacheKey })`**：统一 resolve 链。
4. **CI 性能预算**：warmup 后对 `/api/dashboard-session` 等设 P95 耗时阈值（现仅有结构测试，无耗时门禁）。
