# 规则一览与延期提醒规则

## 文档目的

本文用于沉淀空间视觉项目中台的运营规则、提醒规则和后续开发规则。它不是普通说明文档，而是后续后台判定、前端提醒和 Agent 建议的业务口径来源。

这次先记录「延期提醒规则」的第一版思路。完整规则会按公司项目阶段拆为三部分：

- 硬装提醒规则
- 软装提醒规则
- 摆场提醒规则

本文当前重点仍是硬装 Deadline；项目阶段与下一提醒已沉淀为统一阶段提醒口径，供项目详情、下钻弹窗、团队待处理队列和预计算读模型复用。软装与摆场暂不新增独立 Deadline 公式，但摆场开始、摆场结束和下游事实反推阶段已作为当前生效口径。

硬装 Deadline 分三层维护，不要求在 README、公司背景或前端 HTML 里复制完整规则正文：

| 层级 | 权威位置 | 角色 |

根目录 `README.md` 与 `公司情况与业务环境.md` 只保留高层语境和指向本文的链接。规则仍未确认时，不写成已生效逻辑。

## 运营背景

项目立项和开会时间通常在创建项目时填写，过程中日期可能会有波动，但这些波动不能影响最终开业时间。只要内部阶段推进影响最终开业边界，就需要被识别为延期。

过去系统大量依赖外部表单中已经填写的节点数据去判断延期，导致一个明显问题：如果相关人员没有填写、晚填、乱填节点信息，系统就可能无法识别真实延期。后续规则调整的核心目标，是把关键 Deadline 变成后台硬规则，由系统根据项目阶段、基准日期、面积店态和施工图终审窗口自动生成应完成时间、提醒时间和兜底延期判定，不再让表单缺失本身成为规避判定的空间。

这套规则的管理意图是：

- 把延期责任从「有没有填表」改为「是否在项目阶段硬截止内完成」。
- 把提醒从散乱的状态提示，升级为围绕项目阶段 Deadline 的行动卡点。
- 把延期记录拆细，能看出是启动超时、初稿返回超时，还是最终阶段完成超时。
- 把施工图等外包节点从设计师主责完成口径中拆开，但保留公司项目进度记录。

## 公司项目阶段口径

以下规则都按公司项目阶段口径讨论，不按单个表单字段的偶然填写情况讨论。

- 「提醒」是推动动作的运营提示，不一定等于延期。
- 「延期判定」在钉钉表单已填写时优先采用表单；表单为空时，再由后台 Deadline 规则兜底，不能因为字段缺失而消失。
- 「责任记录」用于说明哪个节点发生过超时，例如延期启动施工图、初稿返回超时。
- 「阶段延期完成」只在该阶段最终完成时间超过阶段终审或完成截止时成立。
- 「开业边界」仍是最终管理边界，内部阶段日期不得反向改写最终开业时间。

## 公司阶段判定和真实项目完成判定

系统里必须长期区分三件事：公司阶段判定、设计责任闭环、真实项目完成判定。它们都叫「完成」时容易混淆，但运营含义不同，后续规则、看板、提醒和 Agent 都必须分开使用。

### 1. 公司阶段判定

公司阶段判定用于首页阶段跑道、项目列表排序、下一提醒和公司协同进度展示。它回答的是：项目当前在公司流程里推进到哪一步。

规则：

- 公司阶段可以根据硬装、软装、施工图、采购、摆场等节点证据向后推进。
- 公司阶段按“最远下游事实”判定：下游节点已经发生时，不能因为上游字段漏填而卡回早期提醒；上游漏填只进入数据缺口提示。
- 公司阶段允许展示并行状态，例如平面方案完成后，施工图外包和点位设计可以同时进入推进视图。
- 公司阶段进入「施工图」「采购」「摆场」「闭环」等后续阶段，不等于项目真实完成。
- 公司阶段只服务运营调度，不得直接作为真实项目完成、绩效完成或最终交付完成的唯一证据。
- 团队工作完成情况里的「项目总闭环」属于公司流程闭环统计口径：常规项目硬装或软装任一轨道精确闭环即计入；睡眠店 / 仅硬装项目按硬装单轨闭环。单轨闭环时详情页提示另一轨道可能漏填写或未更新，但不影响闭环统计。
- 暂停 / 取消是项目停止推进状态，硬装或软装任一轨道命中即按停止项目处理，避免另一轨道漏填导致继续进入推进池。
- 暂停是当前状态，不是历史标签。当前暂停项目暂不进入年度进店、负责人当前负载、团队进行中、超时提醒和风险队列；曾暂停但当前恢复的项目按原始 `startDate` 和当前责任盘重新参与统计。
- 取消是终态，不存在恢复推进。任一轨道或项目状态命中取消 / 已取消 / 关闭 / 已关闭时，项目进入“暂停/取消”归类，并提示“项目已取消”。

### 1.1 统一阶段提醒口径

项目当前阶段与下一提醒只能由一套口径表输出。可执行位置为 `public/domain/project-stage-reminder-rules.mjs`；其他页面不得重新写一套“待软装 / 待采购 / 待摆场 / 待闭环”的分支。

生效原则：

- 主阶段取最远下游事实，优先级从闭环、摆场结束、摆场中、采购完成、采购中、产品清单接收、软装方案、点位、施工图、平面、复尺、上会向前排列。
- 主提醒只表达当前最该推进的动作，例如“等待摆场结束”“项目待闭环”；字段漏填不再直接替代主提醒。
- 数据缺口单独输出，例如“软装完成情况待补填”“摆场开始时间待补填”，供详情页或运营核对使用。
- 暂停、取消和已闭环属于项目状态优先级，先于普通推进提醒。
- 后台硬装 Deadline 提醒仍可作为规则提醒输出，但不得覆盖统一阶段表对当前推进动作的判断。

摆场字段当前口径：

| 事实 | 字段 | 阶段含义 | 下一提醒 |
| --- | --- | --- | --- |
| 摆场开始 | `摆场开始时间`、`摆场时间`、`现场摆场时间` | 填写即进入“摆场中”，不要求当前日期晚于该日期 | 等待摆场结束 |
| 摆场结束 | `摆场文件发出时间(项目群）`、`摆场文件发出时间（项目群）` | 文件已发出即默认现场摆场结束 | 项目待闭环 |
| 采购完成但未摆场 | `采购完成情况` 已完成，且无摆场开始 / 结束事实 | 待摆场 | 待摆场 |
| 只有摆场文件，无摆场开始 | 摆场文件已发出，摆场开始为空 | 仍按摆场结束推进 | 项目待闭环；同时提示摆场开始时间待补填 |

### 2. 设计责任闭环

设计责任闭环用于判断某个岗位或责任线是否完成了自己的主责交付。它回答的是：这个人或这条责任线该交付的内容是否已经闭环。

规则：

- 硬装设计责任从 `平面开始时间` 开始，到 `躺平内部审核结束时间` 或后续明确的硬装方案完成证据为止。
- 施工图通常属于外包或公司后续协同记录，不计入本公司硬装设计师主责完成口径。
- 软装责任拆为点位设计和方案设计两段，点位完成不等于软装方案完成，软装方案完成也不等于公司项目完成。
- 设计责任闭环后，项目仍可能处于施工图、采购、摆场或开业前协同阶段。

### 3. 真实项目完成判定

真实项目完成判定用于最终完成统计、项目关闭、复盘归档和开业边界判断。它回答的是：这个项目是否已经完成最终交付，可以从推进项目中退出。

规则：

- 真实项目完成必须看公司最终交付边界，不能只看某个阶段字段含「完成」。
- 只完成硬装平面、施工图终审、点位设计、软装方案、采购或摆场中的任一单点，都不能单独判定整个项目真实完成。
- 如果公司阶段已经推进到后续协同，但最终交付或开业边界仍未闭合，项目仍应留在推进项目或待闭环项目中。
- 不得把公司阶段推进当成项目真实完成；不得用真实项目完成去反向覆盖阶段过程中的超时记录。

### 4. 使用原则

后续所有规则应按用途选择判定口径：

| 使用场景 | 应使用口径 | 不应使用 |
| --- | --- | --- |
| 首页阶段跑道 | 公司阶段判定 | 真实项目完成判定 |
| 下一提醒 | 公司阶段判定 + 后台 Deadline | 单个完成字段 |
| 个人或团队责任 | 设计责任闭环 | 公司最终闭环 |
| 最终延期状态 | 钉钉表单填写的延期情况；表单为空再用后台 Deadline 兜底 | 用规则覆盖已填写表单 |
| 延期过程记录 | 阶段 Deadline 和责任记录 | 最终完成结果直接覆盖 |
| 项目总完成 / 退出推进池 | 真实项目完成判定 | 公司阶段文本或单点完成字段 |

## 小组页待处理 Top5 队列规则

团队工作完成情况页中的「紧急项目待处理 Top5」「非紧急项目待处理 Top5」两列，用于在当前团队责任盘内快速拉出仍需推进的项目。可执行逻辑以 `src/backend/teamWorkCompletionReview.mjs` 为准。

### 1. 入选范围

项目须同时满足：

- 命中当前看板上下文（直营 / 加盟 / 全部）。
- 已关联到当前团队 roster 成员，未关联团队的项目不进入队列。
- 公司流程未闭环：常规项目硬装或软装任一轨道进入「闭环」即退出；睡眠店按硬装单轨闭环判断。
- 硬装或软装进度已有推进证据，且项目未处于暂停、取消或未开始状态。

### 2. 紧急 / 非紧急分组

分组只读取人工优先级标记，不用于公司阶段、真实完成或 Deadline 判定。

- 读取 `项目状态`、`状态`、`紧急程度`、`项目优先级`、`优先级`，以及原始字段名含「状态 / 紧急 / 优先」的单元格。
- 文本含「紧急」，且不含「不紧急」「非紧急」「一般」→ 进入紧急队列。
- 含「一般」「不紧急」「非紧急」，或未填写优先级 → 进入非紧急队列。

### 3. 交付窗口与排序

- 启动日：`启动时间` / `启动日期` / `开始日期`，回退项目 `startDate`。
- 目标日：优先 `商场交付时间` / `商场交付日期` / `商场交付`；缺失时回退 `计划开业时间` / `计划完成日期` / `截止日期` 或项目 `dueDate`。
- 窗口天数 = 目标日 − 启动日（自然日）。任一端缺失时记为「日期待核对」，排序时靠后。
- 各队列按窗口天数升序排列；同天数按目标日、项目名排序；只取前 5 条作为 Top5，角标展示队列全量待处理数。

### 4. 前端展示

- 行标题：项目名称；副标题显式展示 `店态：...` 与 `当前阶段：...`，当前阶段来自公司阶段（硬装 / 软装进度拼接）；缺失时展示待核对。
- 右侧：显式展示 `交付窗口：...`（紧急列高亮）与 `周期：启动：... → 目标：...`；任一端缺失时只标记对应日期待核对。
- 紧急列在待处理数大于 5 时可展开查看全部；非紧急列只展示 Top5。

## 负责人责任身份与数据通道切分

负责人规则已经进入当前实现口径。系统长期区分「负责人自然人」「责任身份」「责任专业」和「业务范围」，不得把姓名显示、专业归属和直营/加盟范围混成同一个统计 key。

生效规则：

- 负责人自然人仍来自钉钉和本地原始人员数据，例如 `杨锦帆`；本地系统不改钉钉、不写回钉钉，也不把钉钉原始姓名替换成前端显示名。
- 责任身份必须使用稳定技术 id，不能直接用显示名或自然人姓名作为统计 key。当前拆分为 `resp-017` = `杨锦帆（硬装）`、`resp-018` = `杨锦帆（软装）`。
- `杨锦帆（硬装）` 的 `sourceName` 仍为 `杨锦帆`，只匹配 `CD负责人`、`硬装负责人` 等硬装负责人槽位。
- `杨锦帆（软装）` 的 `sourceName` 仍为 `杨锦帆`，只匹配 `VM负责人`、`软装负责人` 等软装负责人槽位。
- 创意团队成员可跨硬装、软装承担组长或设计协作责任，当前先落地马琳琳、熊亮、周俊彤三人；自然人归属记录为创意组长，统计层按责任槽位拆成硬装 / 软装责任身份。
- 创意团队硬装身份包括 `马琳琳（硬装）`、`熊亮（硬装）`、`周俊彤（硬装）`，只进入 `CD组长`、`CD设计师` 等硬装槽位。
- 创意团队软装身份包括 `马琳琳（软装）`、`熊亮（软装）`、`周俊彤（软装）`，只进入 `VM组长`、`点位设计师`、`VM设计师` 等软装槽位；`摆场设计师` 是中性任务责任，不用来反推硬装或软装身份。
- `摆场设计师` 字段写了姓名时，只表示该项目摆场责任由该人承担；公司没有单独摆场设计师岗位。团队负载校验应先看该姓名是否已在本地维护为设计师，或是否在全量项目的 `CD设计师` / `点位设计师` / `VM设计师` / `软装设计师` 字段出现过；有证据则保留摆场统计，不计入当前平面负载。若只出现在 `摆场设计师` 字段且没有设计师证据，进入数据待核对，不反推 CD / VM / 点位身份。
- 项目责任字段是正式人员来源。同步时如果 `CD组长`、`VM组长`、`CD设计师`、`VM设计师`、`点位设计师`、`摆场设计师` 等明确槽位出现本地 SQLite 人员库没有的姓名，应自动补入 active 正职人员；CD / VM / 点位按来源槽位建立硬装 / 软装通道，摆场只建立中性任务事实与摆场责任，不单独证明设计师专业。
- `直营` / `加盟` 只能来自 `组别` 等明确范围字段，不得用负责人姓名反推项目范围。
- `硬装` / `软装` 只能来自责任身份或负责人槽位，不得用硬装/软装进度字段、店态或个人习惯反推责任专业。
- 多身份负责人仅命中 `负责人` 总列、但 `CD负责人` / `VM负责人` 缺失时，进入待核对通道，不进入明确主指标。
- 单一身份负责人仅命中 `负责人` 总列时，可生成推断候选，但默认仍不得静默混入明确主通道。

前端和接口边界：

- 负责人选择器展示责任身份，选中值使用 `identityId`；显示文案、自然人姓名、专业和范围只是属性。
- 负责人月度盘和负责人复盘按责任身份过滤项目：硬装身份只看硬装槽位，软装身份只看软装槽位。
- 同一项目同时命中 `杨锦帆（硬装）` 和 `杨锦帆（软装）` 时，责任身份层可分别计入；看板总量和自然人汇总层按项目 id 去重。
- 同一创意团队成员同时命中 CD / VM 组长或设计师槽位时，责任项按槽位分别进入硬装或软装身份；自然人汇总层仍按项目 id 去重。
- 数据异常、槽位缺失和待核对通道只在本地展示或记录，不反向推送到钉钉。

完整业务备忘见 [人员主数据与责任身份路由契约](../contracts/personnel-and-responsibility-routing.md)。

## 硬装提醒规则

### 0. 时间计算口径

硬装 Deadline 中所有 `Y + N` 都按中国日历工作日计算，不按自然日计算。

规则：

- `Y` 是复尺时间，`Y + 1` 表示复尺日之后的第 1 个中国工作日。
- 系统计算时自动剔除法定双休和国务院办公厅公布的节假日。
- 调休上班日重新计入工作日，例如周六、周日若被官方安排为上班日，则可以成为 `Y + N` 的落点。
- 当前本地日历先记录 2025、2026 年中国法定节假日和调休工作日，文件为 `data/rules/china-workday-calendar-2025.json`、`data/rules/china-workday-calendar-2026.json`。
- 后续跨到新年度时，必须先补对应年度日历，再运行本地规则计算脚本，避免自然日误判。

维护要求：

- 规则页和本文里的矩阵仍写 `Y + N`，但它的含义固定为「第 N 个中国工作日」。
- 本地计算记录由 `scripts/calculate-hard-decoration-deadlines.mjs` 生成，输出到 `data/rules/hard-decoration-deadline-records.json`；该文件同时包含项目级 `records`、设计师级 `designerRecords` 和设计师汇总 `designerSummary`。

### 0.1 延期状态优先级

最终延期状态与系统提醒、效率分析分开处理。

规则：

- 钉钉表单填写的延期情况优先。只要 `硬装方案情况`、`硬装方案情况（每周五刷新）` 或后续明确的延期结果字段已有填写，最终延期状态优先采用该表单结果。
- 表单为空或未写时，才使用后台 Deadline 规则兜底判断延期，避免未填写表单导致项目逃过判定。
- 后台 Deadline 规则不得覆盖已经填写的钉钉表单延期情况；若规则结果与表单结果冲突，应进入后续稽核或数据核对视图，而不是直接改写最终状态。
- 提醒仍按后台 Deadline 规则生成。即使表单已经写了准时或延期，提醒卡点仍可按系统规则推进，后续提醒策略再单独设计。
- 效率判断继续按规则计算。无论表单最终延期状态如何，系统仍记录实际启动顺延后的效率 KPI，例如「延期完成但效率OK」或「效率超时」。

### 1. 复尺后启动平面方案

一个硬装项目只要填写了复尺时间 Y，就进入硬装平面方案启动约束。

规则：

- 复尺时间 Y 存在后，必须在 `Y + 1` 的期限内启动项目。
- 这里的 `Y + 1` 理解为复尺后第 1 个中国工作日的启动窗口。
- 超过该启动窗口仍未启动平面方案，判定为「超时启动」。
- 填写复尺信息时，系统应立刻产生一次提醒：尽快启动平面方案。

注意：当前运营中，设计师不一定准时补填外部表单，其他页面也会有字段补全提醒。为了避免提醒堆叠，复尺后的即时提醒是否进入主提醒队列、在什么阶段展示、是否与字段补全提醒合并，后续再单独定。本文先确认这个硬规则存在。

### 2. 平面方案截止

填写复尺时间 Y 后，系统应根据项目面积对应的店态计算 `Y + X`，作为必要的平面方案截止时间。

规则：

- `X` 已按面积对应店态拆成固定矩阵。
- 方案快截止提醒时间按平面/躺平截止时间倒退 2 个工作日生成。
- 施工图启动时间按平面/躺平截止时间往后 1 个工作日生成。
- 施工图预计截止时间按施工图启动时间继续顺延，mini店到超体店依次增加 3、4、5、6、7、6、7 个工作日。
- 商场审核开始和施工图预计截止时间按同一天记录。
- 终审提醒默认按商场审核开始时间往后 3 个工作日生成。
- 商场终审结束按商场审核开始时间继续顺延，mini店到超体店依次增加 5、6、7、7、7、8、8 个工作日。
- `Y + X` 是平面方案应完成的硬截止。
- 在 `Y + X` 前，应设置一次「平面方案要结束」的提醒节点。
- 超出 `Y + X` 仍未完成平面方案时，系统生成规则侧延期结果。
- 最终延期状态仍按「钉钉表单优先、表单为空再规则兜底」处理；规则侧结果同时用于提醒、效率分析和后续稽核。

#### 平面方案效率判断

延期完成和效率判断必须分开记录。

规则：

- 原始截止判定：平面方案是否延期完成，仍看复尺时间 `Y` 生成的原始 `Y + X` 截止日。
- 原始 Deadline 不因超时启动自动顺延；只要实际平面完成时间晚于原始 `Y + X`，就记录「延期完成」。
- 效率判断：系统同时计算平面方案的工作日预算，即 `平面/躺平设计截止 offset - 平面/躺平设计启动 offset`。
- 例如 mini店：≤300㎡，平面启动为 `Y + 1`，平面截止为 `Y + 6`，效率预算就是 `6 - 1 = 5` 个中国工作日。
- 如果项目超时启动，但按实际平面启动日顺延 5 个中国工作日内完成，则项目仍按原始 Deadline 记录「延期完成」，同时 KPI 记录「延期完成但效率OK」。
- 如果按实际启动日顺延后的工作日预算也被超出，则效率判断记录为「效率超时」。
- 这个效率判断只用于 KPI 和责任分析，不得反向覆盖原始 Deadline 的延期事实。

#### 面积与店态 Deadline 矩阵

可执行矩阵以 `src/backend/hardDecorationDeadlineRules.mjs` 的 `HARD_DECORATION_DEADLINE_MATRIX` 为准；下表为人类可读副本，变更时须与代码常量同步。

以下所有节点都以复尺时间 Y 为基准，且 `Y + N` 统一表示第 N 个中国工作日。面积单位为平方米；`mini店：≤300㎡` 覆盖 `面积 <= 300`，后续中间档按左闭右开理解，例如 `450～650㎡` 表示 `450 <= 面积 < 650`。`超体店：1500～2000㎡` 表示 `1500 <= 面积 < 2000`；`超体店：2000㎡以上（暂按1500～2000㎡）` 覆盖 `面积 >= 2000`，时间暂按 1500～2000㎡ 超体店执行。

| --- | --- | --- | --- | --- | --- | --- | --- | --- |

| 中小店：300～450㎡ | Y + 1 | Y + 4 | Y + 6 | Y + 7 | Y + 11 | Y + 11 | Y + 14 | Y + 17 |
| 中店：450～650㎡ | Y + 1 | Y + 7 | Y + 9 | Y + 10 | Y + 15 | Y + 15 | Y + 18 | Y + 22 |
| 中大店：650～800㎡ | Y + 1 | Y + 9 | Y + 11 | Y + 12 | Y + 18 | Y + 18 | Y + 21 | Y + 25 |
| 大店：800～1000㎡ | Y + 1 | Y + 13 | Y + 15 | Y + 16 | Y + 23 | Y + 23 | Y + 26 | Y + 30 |

### 3. 施工图启动、初稿与终审

硬装平面方案结束后，进入施工图链路：

```text
平面方案结束 -> 启动施工图 -> 初次施工图反馈 -> 商场审核 -> 施工图终审
```

施工图阶段需要拆成几个不同类型的节点：

| 节点 | 作用 | 提醒 | 判定 |
| --- | --- | --- | --- |
| 施工图启动 | 平面方案结束后开始进入外包施工图 | 是否提醒待后续确认 | 超时则记录「延期启动施工图」 |
| 初次施工图反馈 | 外包返回初稿 | 不做提醒 | 超时则记录「初稿返回超时」 |
| 商场审核 | 商场审核与修改反馈 | 以内部卡点控制 | 超出规定节点则记录风险或超时 |
| 施工图终审 | 施工图链路最终结束 | 终审截止前设置一次提醒卡点 | 超出终审截止仍未完成，则施工图阶段延期完成 |

这里的关键点是，施工图完成审核时间记录的是终审结束时间。规则加入后，即使相关人员没有及时更新该时间，系统也应能根据施工图终审截止识别项目已经进入超期风险或延期状态。

### 4. 商场审核窗口

商场审核不是只看商场第一次反馈，而是要把审核、修改、再次提交和终审结束放进同一个公司规定窗口中。

例如：公司规定商场必须在 5 天内审核完毕，则这 5 天应包含内部修改和二次提交时间。实际运营上，同事最好提前拿到修改反馈并处理，确保在规定天数内完成终审。

因此，商场审核规则后续需要同时记录：

- 商场审核开始基准日。
- 公司规定审核结束日。
- 内部建议反馈日。
- 二次修改应提交日。
- 施工图终审截止日。

当前先确认原则：整个施工图链路的最终结束，以施工图终审完成为准。

### 5. 外包节点的责任边界

施工图通常由外包承担，因此施工图初稿、外包返回等节点暂不做提醒，只做延期判定和责任记录。

责任制拆分规则：

- 如果平面方案准时完成，但施工图启动超时，记录「延期启动施工图」。
- 如果施工图初稿返回超过外包约定时间，记录「初稿返回超时」。
- 如果虽然启动超时或初稿超时，但施工图最终仍在终审截止内完成，则施工图阶段不判定为「延期完成」。
- 如果施工图终审超过最终截止，则施工图阶段判定为延期完成。
- 施工图阶段彻底结束前，也就是终审截止前，需要有一个提醒卡点，用于提醒项目即将触碰最终施工图边界。

换句话说，施工图阶段要同时保留「过程超时记录」和「最终阶段延期完成」两个层级，不能把它们混成一个结果。

## 待定参数

以下参数还没有最终定稿，不应写死进业务判定：

- 面积为空或面积明显异常时，系统应进入人工确认还是使用默认兜底档。
- 施工图启动截止：从平面方案结束后几天内必须启动。
- 外包初稿返回截止：按店态、面积或外包约定固定天数计算。
- 商场审核窗口：不同商场是否统一按公司内部规定天数，还是有商场类型差异。
- 终审提醒提前量：终审截止前几天进入提醒队列。
- 即时提醒去重：复尺后启动提醒是否与字段补全提醒合并。

## 软装提醒规则

当前先落地阶段提醒口径，不新增软装 Deadline 公式。

已确认原则：

- 点位、软装方案、产品清单、采购、摆场等下游事实可以推进公司阶段。
- `软装完成情况` 漏填时，如果已经出现采购或摆场事实，主提醒不得停在“待软装完成情况”；漏填只作为数据缺口。
- `软装完成情况` 明确未完成时，若没有更远下游事实，仍停留在软装方案阶段；若已有采购或摆场事实，则按下游事实推进，并保留上游状态冲突提示。

后续若补软装 Deadline，建议按公司项目阶段拆为：

- 点位设计启动与完成。
- 软装方案启动与完成。
- 产品清单接收。
- 采购推进。
- 软装方案完成与公司项目层面软装设计完成的关系。

## 摆场提醒规则

当前先落地阶段提醒口径，不新增摆场 Deadline 公式。

已确认原则：

- `摆场开始时间` 是事实开始时间，填写即进入“摆场中”，主提醒为“等待摆场结束”。
- `摆场文件发出时间(项目群）` / `摆场文件发出时间（项目群）` 是摆场结束事实，填写即默认摆场结束；项目未闭环时主提醒为“项目待闭环”。
- 摆场文件已发出但摆场开始为空时，不阻断阶段推进；系统应提示“摆场开始时间待补填”。
- 项目闭环、暂停和取消仍按项目状态优先级处理，不用摆场字段反推最终项目完成。

后续若补摆场 Deadline，建议按公司项目阶段拆为：

- 采购到货或可摆场确认。
- 摆场准备。
- 现场摆场。
- 摆场交付。
- 闭环完成。

## 后续开发落点

后续实现时，建议把规则拆成三层：

- 规则配置层：保存 `Y + 1`、`Y + X`、施工图启动期限、终审期限等硬规则和版本。
- 项目计算层：根据项目基准日期和规则版本生成 deadline、reminder 和 delay records。
- 展示与 Agent 层：只读取计算结果，不直接用外部表单缺失状态替代后台判定。

这样可以保证后续继续讨论软装、摆场或调整 X 公式时，只更新规则配置和计算逻辑，不需要在多个页面里重复写判断。
