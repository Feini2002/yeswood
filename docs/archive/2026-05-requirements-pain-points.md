# 需求痛点与运营看板需求记录

> 记录日期：2026-05-27
>
> 文档状态：需求记录稿。本文只记录业务现状、痛点、看板目标和待确认问题，不作为开发方案，也不直接规定前端最终设计。

## 一句话结论

本系统要从“钉钉 AI 表格的外部项目看板”进一步明确为“空间视觉部本地运营看盘入口”：钉钉项目记录表负责初始化录入和外部事实输入，本系统负责重新组织、清洗、沉淀本地项目主数据、统计和展示，让部门能看清整体运营、直营与加盟分板块情况，以及主管 / 组长负责范围内的项目状态。

## 用户补充的关键事实

1. 项目记录表类似 Excel，是钉钉侧项目初始化录入和外部事实输入入口。
2. 钉钉里已有多个看盘和仪表盘，但这些截图只帮助理解业务大概需要什么，不能作为本系统前端展示形式的建议。
3. 截图内容来自钉钉表单和钉钉仪表盘，其布局、卡片、图表、表格样式都不构成本系统的 UI 参考。
4. 本系统前端需要展示整个部门的整体运营情况，并支持进一步拆分为直营店、加盟店看板，以及按主管 / 组长负责店铺范围查看各种项目情况。
5. 与总监沟通后确认，现有仪表盘虽然有数据，但对不敏感于数据的人不够直观，需要加入 AI 总结，把数字转成管理者能快速理解的运营判断。
6. AI 总结不应只有一个总总结。更合理的结构是：每个小看盘先生成对应的小总结，系统整体总结再基于多个小总结和关键指标做二次分析。
7. 人员项目负载不能只看项目数量。小团队承接的店铺类型、项目难度、面积、业态不同，最终会影响设计时间和真实负载。
8. 进店量统计需要和小团队负载、项目延期、排班、业态难度、面积等信息联动分析，输出管理建议，而不是只展示月度柱状图。

## 现状判断

当前钉钉项目记录表承担了项目初始录入和原始协作记录职责。它适合持续录入、维护字段、保留原始业务信息，但不适合直接承担公司级或部门级运营看板职责，也不适合作为本地最终业务口径的唯一来源。

钉钉后台已经能按不同视角做统计，例如部门总盘、直营项目、加盟项目、管理者月度看盘等，但这些视图更像原始数据上的临时统计层。对于本系统而言，真正有价值的是这些视图背后的业务问题：项目总量有多少、哪些还在推进、哪些延期、不同店铺类型的压力在哪里、负责人和组长的负载是否均衡、哪个阶段形成堆积。

因此，本系统前端不应该复刻钉钉样式，而应该把钉钉数据重新整理成更适合日常管理、复盘和追踪的运营界面。

## 核心痛点

### 1. 录入入口和分析入口混在一起

钉钉项目记录表是初始化录入入口，这一点需要保留。但当管理者要看全局运营时，Excel 式宽表和钉钉默认仪表盘会让信息分散，难以快速判断当前部门压力、风险和优先级。

本系统需要承担“本地运营分析和最终口径维护入口”的角色。本地可以维护项目状态、风险、负责人归属、进度口径、备注等字段，但不写回钉钉。

### 2. 看盘视角分散

截图中可以看到多个看盘方向：项目统计总盘、直营项目统计、加盟项目统计、管理者月度看盘、VM 加盟专用等。它们说明业务确实需要按不同维度切换观察，但当前入口分散、信息结构不统一。

本系统需要形成统一总入口，再通过清晰的分层进入不同业务看板。

### 3. 直营和加盟需要分开看

部门总盘需要展示整体运营情况，但直营店和加盟店的项目结构、管理方式、延期压力、负责人分布可能不同。钉钉后台数据已经区分直营 / 加盟，本系统读取真实字段后应能识别并分别统计。

直营 / 加盟不只是筛选条件，也应该成为一层稳定的运营视角。

### 4. 主管 / 组长需要看到自己负责的店铺情况

除了部门总览，业务还需要按主管、组长或负责人查看其负责范围内的项目情况。这个视角不是简单看某个人名下有多少项目，而是要能回答：

- TA 负责的店铺目前有多少项目。
- 哪些项目处于推进中、未开始、已完成或延期。
- 哪些阶段形成积压。
- 新店 / 老店、常规店 / 下沉店、不同业态或店态的分布如何。
- 是否存在需要优先介入的风险项目。

### 5. 纯数据仪表盘缺少“管理解释”

现有仪表盘已经能看到项目总数、完成量、运转中项目、延期项目、人员负载、进店量等数据，但这些数据需要使用者自己理解和判断。对于总监或管理层来说，单纯数字不一定能直接回答“现在是否健康”“哪里异常”“谁压力大”“下一步该关注什么”。

本系统需要在数据旁边增加 AI 总结层，把分散指标转成更接近管理语言的判断，例如：本月压力主要来自哪些团队、哪些业态或店态拉高了工作量、延期风险集中在哪里、是否需要调整排班或关注某些负责人。

### 6. 人员负载统计缺少难度权重

截图中的“项目负责人项目总量”和“各组负责人项目总量”能反映数量，但数量不等于真实负载。不同小团队承接不同类型店铺，一个项目的真实工作量可能受以下因素影响：

- 店铺类型不同，例如精品店和普通店的设计复杂度不同。
- 店铺面积不同，面积越大通常设计、复尺、方案、施工图和沟通成本越高。
- 项目业态不同，不同业态可能对应不同设计难度和交付周期。
- 新店、老店、翻新店、下沉店、常规店等店铺属性不同，工作内容和协调成本不同。
- 项目阶段不同，处于方案、施工图、摆场等阶段时，对团队时间占用不同。

因此，人员负载应从“项目数量负载”升级为“加权工作量负载”。前期可以先记录需求和字段关系，后续再确定具体权重模型。

### 7. 进店量需要和负载、延期、排班一起看

“各组月度进店量”能看出某个月进入项目的数量变化，但它本身不能解释团队是否能承接。真正需要关注的是：某个小团队在某月进店量上升时，是否同时存在高难度业态、大面积项目、延期项目集中、排班不足或阶段堆积。

本系统后续应把进店量从单独趋势图，提升为小团队运营分析的一部分。

## 从截图识别到的业务信息

以下内容只用于帮助理解数据结构和业务口径，不能作为前端布局参考。

### 项目记录表可能包含的关键字段

- 项目名称。
- 项目状态。
- 组别。
- 店铺性质。
- 店态。
- 业态。
- 省份。
- 面积。
- 启动时间。
- 计划开业时间。
- 负责人。
- CD 组长。
- VM 或其他协作人员字段。

### 部门总盘中出现的统计方向

- 部门任务总量。
- 全年已完成任务总量。
- 现阶段运转中项目总量。
- 未开始项目总量。
- 直营板块任务数。
- 加盟板块任务数。
- 直营 / 加盟延期项目数。
- 项目负责人项目总量。
- 各组负责人项目总量。
- 各组月度进店量。
- 各店态项目量。
- 按阶段统计项目，例如未开始、待复尺、平面 / 躺平阶段、施工图、软装方案中、摆场中 / 待摆场项目。

### 直营 / 加盟分盘中出现的统计方向

- 直营或加盟任务总量。
- 全年已完成任务总量。
- 现阶段运转中项目总量。
- 未开始项目。
- 新店 / 老店拆分。
- 常规 / 下沉拆分。
- 不同拆分条件下的延期项目数。

### 管理者月度看盘中出现的统计方向

- 按主管或负责人查看新店、老店月度进店量。
- 按超一线、商标店、常规店、下沉店、旗舰店、黑标店、睡眠店、儿童店、维莎店等店态统计项目量。
- 查看未排班、进行中、方案延期、本年方案延期、未闭环延期等项目数。
- 统计全年已完成方案、全年负责总项目、全年闭环项目、现阶段项目量等。

### 新增：AI 总结相关需求

- 部门总盘需要有整体 AI 总结，面向总监或管理层，解释整体运营是否健康。
- 每个小看盘也需要对应小总结，例如直营总结、加盟总结、某个小团队总结、某个主管 / 组长总结、进店量总结、延期风险总结。
- 整体总结不应直接把全部原始数据丢给 AI，而应先由各小模块形成结构化摘要，再进行二次归纳。
- 总结内容应尽量给出判断和建议，而不是简单复述数字。
- AI 总结需要基于本系统后端计算出的指标和规则，避免凭空生成结论。

### 新增：小团队负载相关需求

- 公司部门大约有 8 个小团队。
- 小团队承接的店铺类型不同，团队负载需要按团队维度计算和展示。
- 图中的组长 / 负责人名字，本质上需要支持按小团队负责人或组长聚合。
- 项目负载不能只统计数量，还要考虑店铺类型、项目业态、面积、难度和阶段。
- 负载结果应服务排班和资源协调，例如提示某团队当前承接压力偏高、某类项目集中、某负责人负载异常。

### 新增：进店量与项目运营联动需求

- 进店量统计需要和小团队负载联动。
- 进店量统计需要和项目延期情况联动。
- 进店量统计需要和排班情况联动。
- 进店量统计需要按项目业态拆分。
- 不同业态的难易度、项目面积应挂到团队维度下，形成整体分析。
- 系统应能基于这些关联给出建议，例如关注即将超负荷团队、提前调整排班、优先处理延期风险集中的项目类型。

## 前端看板目标

### 1. 部门总看盘

作为进入系统后的核心页面，优先回答“整个部门现在怎么样”。建议关注：

- 总项目量、推进中、已完成、未开始、延期。
- 直营 / 加盟占比和风险对比。
- 月度新增或进店趋势。
- 阶段分布和积压阶段。
- 店态、业态、省份等结构分布。
- 负责人 / 组长负载排名。
- 需要优先处理的延期或异常项目清单。
- 面向管理层的 AI 整体总结，解释主要变化、风险和建议。

### 2. 直营店看板

聚焦直营店项目的运营情况。它应从部门总盘下钻而来，而不是成为完全割裂的页面。建议关注：

- 直营项目总量和阶段分布。
- 新店 / 老店结构。
- 常规 / 下沉等店铺类型结构。
- 延期项目及延期阶段。
- 负责人和组长负载。
- 月度进店或启动趋势。

### 3. 加盟店看板

聚焦加盟店项目的运营情况。建议关注：

- 加盟项目总量和阶段分布。
- 加盟老店 / 加盟新店等结构。
- 常规 / 下沉等店铺类型结构。
- 延期项目及延期阶段。
- 负责人和组长负载。
- 月度进店或启动趋势。

### 4. 主管 / 组长看板

用于按管理责任范围查看项目。建议支持选择某个主管、组长或负责人后，展示 TA 负责店铺的完整运营情况：

- 负责项目总数。
- 进行中、未开始、已完成、延期。
- 阶段分布。
- 直营 / 加盟、新店 / 老店、店态、业态拆分。
- 月度趋势。
- 风险项目和延期项目明细。
- 基于项目数量、项目难度、面积、业态和阶段的加权负载判断。

### 5. 小团队负载看板

小团队负载看板用于回答“哪个团队现在压力大，压力来自哪里”。它不应只展示负责人项目数量，而应尽量拆出负载来源：

- 团队承接项目总数。
- 团队成员或负责人项目分布。
- 项目按直营 / 加盟、新店 / 老店、店态、业态拆分。
- 项目面积区间分布。
- 不同难度项目占比。
- 当前阶段分布和阶段堆积。
- 延期项目、未排班项目、进行中项目。
- 加权负载分或负载等级。
- AI 小结：说明团队压力来源、是否需要排班调整、哪些项目类型需要优先关注。

### 6. 进店量运营看板

进店量看板不只是按月份统计数量，而是要和团队承接能力、延期风险和排班共同分析：

- 按月统计进店量。
- 按小团队拆分进店量。
- 按直营 / 加盟、新店 / 老店、店态、业态拆分进店量。
- 进店项目面积和难度分布。
- 进店量与当前团队负载对比。
- 进店量与延期项目变化对比。
- 进店量与未排班 / 排班中项目对比。
- AI 小结：说明某月进店压力是否异常、压力集中在哪些团队和业态、是否需要提前协调资源。

### 7. 项目明细与下钻

看板上的关键指标都应能下钻到项目明细。明细需要同时保留本地最终项目字段和钉钉原始来源字段：本地字段用于最终展示和管理口径，钉钉原始字段用于追溯和诊断。前端展示名可以专业化处理，避免直接暴露过长、口语化或带内部提醒的字段名。

## AI 总结层需求

AI 总结层的目标不是替代数据图表，而是帮助管理者更快读懂图表。

### 总结层级

```text
单个指标 / 图表
  -> 小模块结构化摘要
  -> 小看盘 AI 总结
  -> 部门整体 AI 总结
```

其中，小看盘总结可以包括直营、加盟、小团队、主管 / 组长、进店量、延期风险等不同模块。部门整体总结应基于这些小总结做二次分析，形成更高层的运营判断。

### 总结内容方向

- 当前最重要的运营变化。
- 项目量是否异常。
- 延期风险是否集中。
- 哪些团队或负责人负载偏高。
- 哪些业态、店态或面积段带来更高压力。
- 是否存在排班或资源协调建议。
- 哪些项目或团队需要优先下钻查看。

### 总结边界

- AI 总结必须基于系统已经计算出的指标、规则和项目明细。
- AI 总结不能虚构原因，无法判断时应表达为“需要进一步确认”。
- AI 总结应尽量给出管理动作建议，但不直接替代负责人决策。
- AI 总结需要保留可追溯性，最好能说明结论来自哪些指标或模块。

## 加权负载分析需求

加权负载用于解决“项目数量不等于真实工作量”的问题。该需求暂不确定具体公式，但需要先记录影响因素。

### 可能影响负载的因素

| 因素 | 说明 |
| --- | --- |
| 项目数量 | 最基础的负载指标，但不能单独使用 |
| 店铺类型 | 精品、普通、常规、下沉等不同类型可能复杂度不同 |
| 直营 / 加盟 | 管理方式和协作链路不同，可能影响沟通成本 |
| 新店 / 老店 / 翻新店 | 工作内容和复用程度不同 |
| 项目业态 | 不同业态设计难度、标准化程度和交付周期可能不同 |
| 面积 | 面积通常影响方案、图纸、陈列和沟通工作量 |
| 项目阶段 | 方案、施工图、摆场、闭环等阶段占用的人力不同 |
| 延期状态 | 延期项目通常需要额外协调、跟进和复盘 |
| 排班状态 | 未排班或排班不足会影响团队承接风险 |

### 可能的负载表达

- 项目数量负载：直接统计项目数，用于快速理解规模。
- 加权项目负载：按类型、业态、面积、阶段等因素换算成工作量。
- 负载等级：例如正常、偏高、过载，用于管理层快速判断。
- 负载来源拆解：说明负载高是因为数量多、面积大、难度高、延期多，还是某阶段堆积。

### 负载模型边界

负载权重必须后续和业务负责人确认。前期可以先做字段沉淀和口径设计，不急于把权重写死。任何负载分都应定位为运营辅助，不作为个人绩效结论。

## 产品边界

- 钉钉项目记录表仍是初始化录入和外部事实输入入口。
- 本系统可以维护本地最终项目口径，但不新增、不编辑、不删除钉钉数据，也不写回钉钉。
- 钉钉截图只用于理解业务，不作为 UI 设计稿。
- 前端展示形式需要重新构思，不能照搬钉钉卡片、图表和表格。
- 指标必须可解释，关键口径需要写入文档。
- 直营、加盟、主管、组长等视角都应来自真实钉钉字段或明确的字段映射规则。
- AI 总结只做数据解释和运营建议，不替代管理决策。
- 加权负载只做团队运营辅助，不作为个人绩效结论。

## 后续需要确认的问题

1. 直营 / 加盟的准确来源字段是什么？是“组别”“店铺性质”，还是另有专门字段？
2. 主管 / 组长的准确口径是什么？按“负责人”、按“CD 组长”、按 VM，还是按另一套管理归属字段？
3. 如果一个项目有多个负责人或多个协作人，主管 / 组长看板中应该只算主责人，还是每个相关人都计入？本地是否需要维护最终主责人字段？
4. “已完成”“闭环”“全年已完成”的业务口径是什么？哪些项目状态算完成，跨年未闭环项目如何处理？
5. “延期”的统一定义是什么？只按计划开业时间判断，还是需要区分方案延期、施工延期、摆场延期等阶段性延期？
6. 部门总看盘的时间口径是自然年、滚动 12 个月、当前月，还是需要支持自由筛选？
7. 看板刷新频率期望是什么？每天定时同步、手动同步，还是工作时间内更高频同步？
8. 部门约 8 个小团队的准确团队名单、负责人、承接店铺类型分别是什么？
9. 店铺类型、项目业态、面积、店态、新店 / 老店等字段中，哪些是影响工作量的核心因素？
10. 精品店、普通店、不同业态、不同面积段的难度权重是否已有经验口径？
11. 排班数据是否已经在钉钉表中，还是需要来自另一张表或后续人工维护？
12. AI 总结需要多久刷新一次？随数据同步生成，还是进入页面时即时生成？
13. AI 总结是否需要区分阅读对象，例如总监版、组长版、执行负责人版？

## 本次记录对后续开发的影响

后续前端重构时，应先围绕“部门总盘 -> 直营 / 加盟分盘 -> 主管 / 组长责任盘 -> 项目明细下钻”建立信息架构，再决定具体组件、图表和布局。

这意味着下一步不应该直接美化当前页面，而应先确认字段口径、管理视角、AI 总结层级和加权负载口径，再设计新的前端信息结构。

尤其是小团队负载和进店量分析，后续需要从单一图表升级为“数据 + 解释 + 建议”的运营模块。前端最终展示应帮助管理者快速理解：当前压力在哪里、为什么形成压力、应该优先看哪里、是否需要调整排班或资源。
