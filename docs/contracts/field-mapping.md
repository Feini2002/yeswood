# 钉钉原始字段到前端展示映射

## 本次方向调整

当前过渡阶段，前端明细页以钉钉 AI 表格真实返回字段作为来源参考。mock 数据只用于开发兜底，不再作为字段设计依据。

系统升级为 SQLite 本地主数据后，最终项目台账和指标应优先使用本地最终项目字段。钉钉原始字段仍要保留，用于来源追溯、字段诊断和导入映射。

系统仍然保留少量“标准字段”，但它们只服务于指标、筛选、搜索和风险判断。前端明细表不再写死这些标准字段，而是按钉钉原始字段动态生成列。

## 重要补充：原始字段名不等于前端展示名

钉钉字段名是内部协作字段，可以包含括号、提醒、刷新频率、口语化提示或临时说明。这些内容对内部核心人员很方便，但不一定适合直接显示在公司级 SaaS 前端。

因此系统需要区分两层名称：

| 层级 | 说明 | 示例 |
| --- | --- | --- |
| 原始字段 key | 后端精确对应钉钉字段，不能随意改，否则会断开数据映射 | `硬装方案情况（每周五刷新）` |
| 前端展示 label | 面向用户的专业字段名，可以去除内部提示和括号说明 | `硬装方案情况` |

原则：

- 后端取数、原始导入记录、`rawFields` 的 key 必须保留钉钉原始字段名。
- 前端表头可以使用更专业的展示名。
- 展示名变化不能影响后端取数。
- 每一个人工确认过的展示名映射都必须写入本文档。
- 未确认展示名的字段，前端可以暂时使用原始字段名。

当前已确认的展示名映射：

| 钉钉原始字段 key | 前端建议展示 label | 原因 |
| --- | --- | --- |
| `硬装方案情况（每周五刷新）` | `硬装方案情况` | 去除内部刷新提醒，使前端字段更专业 |

## 数据分层

```text
钉钉原始记录
  -> record.fields
  -> SQLite dingtalk_raw_records：保留钉钉原始字段 key
  -> 本地 projects / overrides：形成最终项目字段
  -> field_aliases / fieldCatalog：字段 key + 展示 label
  -> 前端字段展示名映射和项目台账
```

## 后端输出结构

### 每条项目记录

后端每条 `project` 同时包含两类字段。

标准字段：

```json
{
  "name": "用于指标和搜索的项目名称",
  "province": "用于筛选的省份",
  "businessType": "用于筛选的业态",
  "storeStatus": "用于筛选的店态",
  "status": "用于状态统计",
  "owner": "用于负责人负载",
  "progress": 0,
  "dueDate": "计划开业管理边界，用于展示、筛选和开业边界统计"
}
```

钉钉原始字段展示层：

```json
{
  "rawFields": {
    "钉钉字段名": {
      "display": "前端可读值",
      "kind": "string | number | date | object | array | link | boolean | empty | mixed"
    }
  }
}
```

### 字段清单

当前 `GET /api/projects` 返回：

```json
{
  "items": [],
  "fieldCatalog": [
    {
      "key": "钉钉字段名",
      "label": "钉钉字段名",
      "kind": "字段类型",
      "nonEmpty": 123
    }
  ],
  "readOnly": true
}
```

前端使用 `fieldCatalog` 生成明细表表头，并用每条记录的 `rawFields[field.key].display` 填充单元格。

注意：`fieldCatalog.key` 永远应对应钉钉原始字段名；`fieldCatalog.label` 或前端后续的 alias 配置可以用于更专业的展示名。SQLite 升级后，该结构仍可作为来源字段目录，不应替代本地最终项目主表。

## 类型转换规则

| 钉钉返回类型 | 前端显示方式 |
| --- | --- |
| string | 去除前后空格后原样显示 |
| number | 如果像 10 位或 13 位时间戳，显示为日期；否则显示数字 |
| object 且有 `name` | 显示 `name` |
| object 且有 `text` | 显示 `text` |
| object 且有 `link` | 显示 `text (link)`，没有 text 时显示 link |
| array | 每一项按同样规则转换，再用 `、` 拼接 |
| null / undefined | 显示为空，前端表格兜底为 `--` |
| 无法识别的 object | 转为 JSON 字符串 |

## 前端明细页规则

- 表格数据列来自钉钉原始字段名。
- 表头展示可以经过专业化别名处理。
- 列顺序来自后端同步时汇总出的字段出现顺序。
- 单元格显示后端格式化后的 `display`。
- 长文本在单元格内省略，完整值放在浏览器原生 title 中。
- 当前明细页仍然不提供写入控件。后续如增加编辑，只能写入本地 SQLite，不能写回钉钉。

## 前端字段别名维护规则

未来会陆续出现更多内部字段名需要专业化展示，例如：

- 去掉括号中的内部提醒。
- 去掉“不要自己添加”等操作提示。
- 把过长字段名改成更短的业务名。
- 把内部小组习惯用语改成公司统一术语。

维护要求：

1. 不能直接改钉钉原始 key。
2. 不能让前端展示名反向影响后端同步。
3. 每次新增字段别名，都要在本文档的“当前已确认的展示名映射”表中增加一行。
4. 如果后续实现代码层别名配置，配置文件或代码中的映射必须与本文档一致。
5. 字段别名只解决展示专业性，不解决指标口径。指标口径仍需更新 [看板指标与 Profile 契约](./dashboard-metrics.md)。

## 标准字段仍然存在的原因

虽然当前明细表使用钉钉原字段，但看板顶部指标和后续项目台账仍然需要统一字段：

- 项目总数。
- 状态统计。
- 延期判断。
- 负责人负载。
- 地区、业态、店态筛选。
- 平均进度。

因此 `.env` 中的 `DINGTALK_FIELD_MAP_JSON` 仍然保留。它不再决定明细表列名，只决定哪些钉钉字段被导入为项目标准字段。SQLite 升级后，本地覆盖字段优先于该映射导入值。

## 后续维护规则

每当字段展示规则变化时，必须同步更新本文档。例如：

- 某类对象字段不再取 `name`，改取其他键。
- 时间戳判断规则变化。
- 链接字段展示方式变化。
- 前端开始隐藏某些钉钉字段。
- 前端给某些钉钉字段增加专业展示名。
- 指标层标准字段映射发生变化。
