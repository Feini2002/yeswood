# 架构决策记录

> 本文只保留长期有效的架构决策。阶段路线图、当前状态和待办项见 [`../STATUS.md`](../STATUS.md)。

## ADR-001：前后端分离，前端不接触钉钉

决定：前端只请求本系统 API，不接触钉钉。

理由：保护 token，保持安全边界，避免浏览器成为钉钉接口调用方。

## ADR-002：本地 JSON 缓存作为第一阶段数据层

决定：第一阶段使用本地 JSON 缓存。

理由：依赖少，调试快，适合原型和内部验证。

状态：该决策进入历史阶段。系统已升级为 SQLite 本地主数据底座第一版；JSON 缓存保留为兼容快照和回退线索。

## ADR-003：不实现写回钉钉

决定：不实现任何钉钉写入接口。

理由：本地系统可以维护最终业务口径，但不反向污染钉钉原始协作表，也不扩大钉钉 token 的使用风险。

## ADR-004：阶段进度允许本地估算

决定：钉钉阶段枚举可在本地映射为 0-100 展示进度。

理由：看板需要平均进度，但真实表的阶段不是百分比。

限制：估算结果只用于本地展示和运营判断，不写回钉钉，不作为绩效结论。

## ADR-005：前端明细页以钉钉原始字段为来源参考

决定：当前前端明细表按后端 `fieldCatalog` 动态生成列，单元格使用 `rawFields` 中的展示值。

理由：真实钉钉表的字段名和结构是导入来源，需要在过渡期可见。

状态：后续前端升级时，项目台账应逐步以本地最终项目字段为主，钉钉原始字段转为来源参考和字段诊断依据。字段展示契约见 [字段映射契约](./field-mapping.md)。

## ADR-006：人员架构关系由本地配置优先表达

决定：钉钉项目表继续作为项目参与事实来源；负责人团队、组长归属、人员别名、隐藏人员等组织关系由本地人员架构配置优先表达。

理由：钉钉人员字段记录的是项目参与人员，不是组织从属关系。仅凭同项目协作不能准确判断“某组长属于哪个负责人团队”。

限制：本地人员架构配置不写回钉钉，也不应写进钉钉 raw records。人员与责任路由见 [人员主数据与责任身份路由契约](./personnel-and-responsibility-routing.md)。

## ADR-007：SQLite 本地主库作为最终项目数据源

决定：采用 SQLite 作为本地最终项目主数据底座，默认文件为 `data/app.sqlite`。

理由：当前系统低负载、少数人员使用，需要本地编辑、覆盖记录、审计和差异追踪，但不需要复杂多人数据库服务。

限制：数据库访问必须封装，避免业务代码直接依赖具体 SQLite 实现。`data/app.sqlite` 不得提交 Git，详见 [安全边界](./security-boundary.md) 与 [Git 与本地数据规范](../handbook/git-and-data.md)。
