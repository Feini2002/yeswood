# 人员主数据与责任身份路由契约

> 状态：人员主数据来源 + 负责人责任身份与数据通道切分的合并契约。  
> 相关契约：[数据权威与钉钉导入契约](./data-authority.md)、[看板指标与 Profile 契约](./dashboard-metrics.md)、[运营规则正文](../rules/operational-rulebook.md)。

## 背景判断

钉钉 AI 表格当前承担的是项目初始化录入和原始协作记录入口。人员字段来自整个部门的搜索选择，例如负责人、CD 组长、VM 组长等字段记录的是“这个项目出现了哪些人”，不是“这个人归哪个负责人团队管理”。

因此，当前钉钉表格不能准确表达 8 个负责人各自管理哪些组长。仅凭同一个项目中同时出现负责人和组长，只能说明项目协作关系，不能说明组织从属关系。

## 数据源分工

钉钉项目表是项目来源事实：

- 项目名称、地区、业态、店态、状态、进度、日期、延期、风险等项目事实。
- 项目行中出现过哪些负责人、CD 组长、VM 组长、CD 设计师、VM 设计师等参与人员。
- 项目明细页的原始字段展示。

本地人员架构配置是组织关系事实和最终管理口径：

- 哪些人是负责人。
- 每个负责人下面有哪些 CD 组组长和 VM 组组长。
- CD 组长属于硬装线，VM 组长属于软装线。
- CD 设计师属于硬装线，VM 设计师属于软装线。
- 负责人也可能区分硬装 / 软装。
- 人员别名、姓名归一化、展示顺序、隐藏或离职规则。
- 团队架构页面和未来负责人团队视角的归属关系。

优先级：

1. 本地人员架构配置。
2. 钉钉项目数据中真实出现过的人员记录。
3. 系统临时推导逻辑。

本地配置一旦明确某人的角色或团队归属，应优先于钉钉字段推导。钉钉数据只补充“此人在项目中出现过”“参与过哪些项目”“形成多少项目负载”等项目事实。

## 当前实现约束

- 人员情况页默认展示负责人、CD 组组长、VM 组组长。
- 设计师已经进入本地人员主数据，但不进入默认架构总表主视觉；进入“编辑架构”模式后，设计师可在人员编辑台维护。
- 编辑架构模式是本地人员主数据工作台，不是钉钉字段修正页。
- CD 组组长和 VM 组组长允许平调。平调本质是 `position = "lead"` 不变，`discipline` 在 `hard` / `soft` 之间切换。
- 负责人团队配置通过页面内人员选择器维护，不使用原生多选框作为主要交互。
- 保存、放弃草稿等关键动作使用页面内确认面板，不依赖浏览器原生弹窗。
- 保存时以后端人员主档为准重算 `roleGroups`，避免前端草稿中旧角色组残留导致保存失败。

架构总表 UI 可仍将某人只展示在负责人栏；统计层必须按钉钉列分别计数：同人在 `负责人` 与 `CD组长` 列出现则两个 slot 各自 +1，禁止跨列去重。

## 人员主数据

推荐维护独立本地人员主数据：

```text
data/personnel-database.json
```

这份文件是人员主数据种子文件，不应被钉钉同步覆盖。`data/dashboard-cache.json` 只是历史阶段的钉钉项目快照缓存，不适合人工维护组织关系。当前 SQLite 已新增 `personnel_people`、`personnel_role_members`、`personnel_teams`、`personnel_team_members`，前端编辑后以 SQLite 为准，JSON 只作为空库初始化和回退线索。

当前人员主数据已按以下框架搭建：

- 硬装负责人。
- 软装负责人。
- 创意负责人（硬装+软装双负责，当前仅杨锦帆一人）。
- 硬装组长。
- 软装组长。
- 创意组长（创意团队跨硬装+软装责任项）。
- 硬装设计师。
- 软装设计师。
- 负责人责任身份（统计通道，不是钉钉字段）。

## 杨锦帆拆分原则

后台钉钉数据和本地原始人员数据仍只读取 `杨锦帆` 这个自然人姓名，不改钉钉、不写回钉钉，也不把原始姓名替换成前端显示名。

| identityId | displayName | sourceName | discipline | scope | validFrom | validTo |
| --- | --- | --- | --- | --- | --- | --- |
| `resp-017` | 杨锦帆（硬装） | 杨锦帆 | hard | both | 待定 | null |
| `resp-018` | 杨锦帆（软装） | 杨锦帆 | soft | both | 待定 | null |

规则：

- `杨锦帆（硬装）` 只匹配 `CD负责人`、`硬装负责人` 等硬装负责人槽位。
- `杨锦帆（软装）` 只匹配 `VM负责人`、`软装负责人` 等软装负责人槽位。
- 责任身份必须使用稳定技术 id，而不是直接使用显示名作为统计 key。
- 如果未来杨锦帆不再负责硬装，不应修改 `resp-017` 的历史含义；应填写 `validTo` 关闭旧身份，并按新口径创建或启用新的责任身份。

## 创意团队成员拆分原则

- `摆场设计师` 只作为中性任务责任项，不反推创意组硬装或软装身份，也不单独证明设计师专业。

## 数据通道安全边界

- 钉钉仍是项目初始化录入和外部事实输入来源，本地系统不写回钉钉。
- 钉钉原始字段值必须保留原样，尤其是负责人姓名、`组别`、`CD负责人`、`VM负责人` 等字段。
- 本地切分只发生在责任身份解析、指标过滤和前端展示层，不反向改写项目原始字段。
- `直营` / `加盟` 只能来自 `组别` 等明确范围字段，不得用负责人姓名反推项目范围。
- `硬装` / `软装` 只能来自责任专业或字段槽位，不得用店态、常规/下沉或个人习惯反推责任专业。
- 同一自然人拆成多个责任身份后，统计 key 必须使用责任身份 id，不能只用姓名。
- 任何字段缺失、槽位冲突或无法判断责任专业的记录，应进入 `待核对`、`推断候选` 或数据异常通道，不得静默混入明确主通道。

## 同步时的人员入库边界

- `CD组长`、`VM组长`、`CD设计师`、`VM设计师`、`点位设计师`、`摆场设计师` 等项目责任字段出现新姓名时，视为正式人员已经进入项目事实。
- 如果本地 SQLite 人员库没有该姓名，同步流程应自动补入 active 人员，`source` 记录为 `dingtalk-derived`，并保留来源字段。
- 自动补入只使用明确责任槽位判断专业：CD 槽位进硬装，VM / 点位 / 方案设计槽位进软装；摆场槽位只补正式人员事实与任务责任，不反推专业，也不单独证明该人是设计师。
- 团队负载页面校验摆场责任人时，只把已维护设计师，或在全量项目 `CD设计师` / `点位设计师` / `VM设计师` / `软装设计师` 字段出现过的人视为已有设计师证据。
- 只出现在 `摆场设计师` 字段的人进入数据待核对，不进入真实负载。
- 已有本地人员记录不被自动覆盖；本地精准维护仍优先于项目字段自动补入。

## 通道类型

| 通道 | 进入条件 | 是否进入明确主指标 | 前端处理建议 |
| --- | --- | --- | --- |
| 明确主通道 | 业务范围明确，且自然人命中对应专业槽位 | 是 | 正常计入负责人和看板统计 |
| 推断候选通道 | 单一身份负责人仅命中 `负责人` 总列，缺少对应 CD/VM 槽位 | 默认否；如后续业务确认启用，也必须单独标记并可筛除 | 标记 `槽位缺失-按人员配置推断`，供运营核对 |
| 待核对通道 | 多身份负责人仅命中 `负责人` 总列，或槽位无法判断专业 | 否 | 进入数据待核对列表，需补充或修正 CD/VM 负责人 |
| 数据异常通道 | `组别` 缺失、范围不明、负责人槽位冲突、自然人无法归一 | 否 | 在负责人面板或项目明细中提示异常，不写回钉钉 |

## 通道判定顺序

1. 先读取钉钉原始项目事实：项目、`组别`、负责人槽位、硬装/软装进度字段。
2. 用 `组别` 判定业务范围：`直营`、`加盟` 或 `other/待核对`。
3. 用责任槽位判定专业通道：
   - `CD负责人` / `硬装负责人` -> 硬装通道。
   - `VM负责人` / `软装负责人` -> 软装通道。
   - 已知多身份负责人仅 `负责人` 总列命中、拆分槽位缺失 -> 待核对。
   - 单一身份负责人仅 `负责人` 总列命中、拆分槽位缺失 -> 生成推断候选。
4. 用本地人员配置把自然人映射为责任身份。
5. 前端按责任身份展示，指标按责任身份聚合。

## 进度字段角色

硬装/软装进度字段是项目阶段、完成、延期和负载判断的事实输入，不参与责任专业通道归属。

- 不能因为 `硬装项目进度` 有值，就把某个负责人归入硬装通道。
- 不能因为 `软装项目进度` 有值，就把某个负责人归入软装通道。
- 责任专业归属优先看负责人槽位和本地人员配置。
- 进度字段只在通道确定后，用于判断该通道内的阶段、完成、延期、当前负载或公司协同状态。

## 计数与去重

- 责任身份层：`杨锦帆（硬装）` 和 `杨锦帆（软装）` 可分别计入同一个项目，用于展示各自责任线的工作量、延期和完成情况。
- 自然人汇总层：如果展示 `杨锦帆` 这个自然人总项目数，必须按项目 id 去重。
- 看板总量层：直营、加盟、硬装、软装等板块总量按项目 id 去重。
- 责任项或工作量层：如果业务明确要看责任项数量，可以按身份分别计数，但文案必须写清是责任项，不是项目数。
- 前端明细中同一项目命中多个责任身份时，应标记 `多身份命中` 或同等说明，避免用户误以为数据重复。

## 团队下属归属

负责人下属组长和成员的归属来源应是本地人员架构，不是钉钉组织架构，也不能从项目协作字段临时反推。

- 团队归属绑定到责任身份或明确的负责人团队口径，而不是只绑定自然人姓名。
- 苏佳蕾团队默认进入 `直营硬装团队`；其组长和成员主要服务直营硬装责任制。
- 后续补充其他负责人团队时，应同时记录团队的 `discipline` 和 `scope`。
- 如果一个成员存在矩阵协作或多个上级，应先记录为待确认关系；未经确认不得同时计入多个团队当前负载。
- 项目字段里的设计师、组长、协作人只说明项目参与事实，不等于组织从属关系。

## 不得混淆的口径

- 负责人自然人不等于责任身份。
- 直营/加盟不等于硬装/软装。
- 公司阶段不等于团队责任制完成。
- 施工图、产品清单、采购、摆场可以作为公司协同阶段记录，但不得直接计入硬装或软装设计师主责完成口径。
- 点位设计完成不等于软装方案完成。
- 软装方案完成不等于项目真实完成。

## 待确认问题

- 杨锦帆在仅命中 `负责人` 总列、但 `CD负责人` / `VM负责人` 缺失时，当前实现已一律进入待核对；后续如运营确认允许推断，需要单独改规则和测试。
- 其他负责人团队补充后，是否允许某个负责人也拥有双范围或双专业身份。
- 前端负责人选择器是否展示自然人分组，还是直接展示责任身份列表。
- 数据异常通道如何在负责人面板展示，避免被误解为负责人真实负载。

## 开发注意事项

- 不要把 `data/dashboard-cache.json` 或 `dingtalk_raw_records` 当成人员主数据源。
- 新增、修改、隐藏人员时，应优先通过人员情况页写入 SQLite `personnel_*` 表；`data/personnel-database.json` 只作为种子文件和空库回退线索。
- 不要把人员名单流水写进开发说明文档。
- 不要仅凭同项目协作关系生成负责人团队归属。
- 不要仅凭项目中存在硬装 / 软装进度字段来判断负责人属于硬装或软装。
- 不要把多责任身份自然人的 `负责人` 总列命中静默归入硬装或软装个人盘；无法判定 slot 时进入待核对。
- 不要把人员架构关系写回钉钉。
- 新增团队架构接口时，应同时读取钉钉来源数据和本地人员主数据，再在后端合并。
- 前端人员架构页应优先展示本地配置中的团队关系，项目负载、延期、风险等再从钉钉项目事实中补充。
