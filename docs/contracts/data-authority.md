# 数据权威与钉钉导入契约

## 定位

本系统是“空间视觉项目本地数据中台”。钉钉 AI 表格负责项目初始化录入和外部事实输入；本系统负责把钉钉数据同步导入本地，沉淀为 SQLite 项目主数据，并在本地维护最终业务口径。

系统不再被定位为单纯的“钉钉只读增强层”。钉钉是导入源，本地 SQLite 是最终项目数据源。

## 核心目标

- 让项目数据更容易被管理层、项目负责人和协作人员看懂。
- 让项目状态、延期风险、负责人负载、地区分布、业态和店态分布可筛选、可统计、可复盘。
- 建立本地完整项目主数据，允许本地覆盖项目状态、风险、负责人归属、进度口径、备注等最终业务字段。
- 保持对钉钉的最小侵入：不改变钉钉原表结构，不写回钉钉，不把钉钉 token 暴露到前端。
- 为后续前端运营工作台、字段配置、人员维护、历史复盘和差异确认预留清晰边界。

## 数据链路

```text
钉钉 AI 表格
  -> 后端同步导入服务
  -> SQLite 原始导入记录
  -> SQLite 本地项目主数据
  -> 本地覆盖 / 差异 / 审计层
  -> 字段清洗与指标层
  -> API
  -> 前端看板与运营工作台
```

## 数据权威关系

```text
本地 SQLite 最终数据 > 本地覆盖规则 > 钉钉最新导入值 > 系统推导默认值
```

含义：

- 钉钉记录提供初始化和外部事实参考。
- 本地项目主表提供最终展示、筛选和指标口径。
- 本地覆盖字段不能被后续钉钉同步静默覆盖。
- 钉钉值和本地值发生差异时，应进入差异记录或待确认流程。

## 系统边界

前端边界：

- 前端只请求本系统后端的 `/api/*`。
- 前端不直接请求钉钉接口。
- 前端后续可以提供本地编辑交互，但只写入本地 SQLite。
- 前端不得出现任何“写回钉钉”的动作。

后端边界：

- 后端可以获取 token。
- 后端可以调用钉钉 records/list 读取数据。
- 后端必须处理分页，直到 `hasMore=false`。
- 后端可以写入 SQLite、清洗字段、计算指标、记录本地修改审计。
- 后端不得实现钉钉写入接口。

## 同步协议

### token 获取

后端通过环境变量配置 token 获取方式。允许两类方式：

- 传统 AppKey/AppSecret 换取 token。
- 内部 token provider 直接返回 token。

无论哪种方式，token 只存在于后端运行时，不进入前端。

### records/list 读取

后端调用钉钉 records/list 接口读取数据。

分页规则：

```text
首次请求:
  body 包含 maxResults

如果响应 hasMore=true:
  读取响应 nextToken
  下一次请求 body 加入 nextToken

直到:
  hasMore=false
```

如果 `hasMore=true` 但缺少 `nextToken`，同步必须失败并返回错误，不允许静默吞掉数据。

## 标准项目字段

导入后，每条项目记录应尽量映射为以下本地标准字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | string | 钉钉记录 id 或稳定本地 id |
| name | string | 项目名称 |
| province | string | 省份 |
| businessType | string | 业态 |
| storeStatus | string | 店态 |
| status | string | 项目状态；只表示管理优先级，不等同流程阶段 |
| owner | string | 负责人 |
| cdOwner | string | CD/硬装负责人 |
| vmOwner | string | VM/软装负责人 |
| progress | number | 0-100 的本地展示进度 |
| startDate | string | 启动时间；管理边界 / 进店月份统计使用，不参与项目明细下一提醒 |
| dueDate | string | 计划开业时间；管理边界 / 开业边界统计使用，不等同阶段完成日期 |
| riskLevel | string | 高 / 中 / 低 |
| riskNotes | string | 风险说明 |
| updatedAt | string | 最近更新时间 |
| isDelayed | boolean | 是否延期 |
| source | string | 来源标记，例如 dingtalk-ai-table |

## 字段映射

系统采用两层字段映射：

1. 自动解析：同步时扫描钉钉原始列名，按标准字段语义表自动识别。管理员修改括号内备注时，系统仍可通过基础字段名前缀识别。
2. 环境变量覆盖：`.env` 的 `DINGTALK_FIELD_MAP_JSON` 用于强制指定标准字段对应的钉钉列名；仅当该列名仍存在时生效，优先级高于自动解析。

匹配优先级：

```text
.env 精确命中
  -> SQLite 缓存绑定
  -> 规则自动解析（归一化 / 精确 / 前缀 / 关键词消歧）
```

自动解析实现见 [`src/backend/fieldResolver.mjs`](../../src/backend/fieldResolver.mjs)。解析结果会写入 SQLite `field_source_bindings` 表，供后续同步复用。

原始字段到前端展示名的详细规则见 [字段映射契约](./field-mapping.md)。

## 标准字段语义表

| 标准字段 | 识别规则 | 说明 |
| --- | --- | --- |
| name | `项目名称`、`门店名称` 前缀匹配 | 括号备注变化不影响识别 |
| province | `省份` 精确匹配 | |
| businessType | `业态` 精确匹配 | |
| storeStatus | `店态` 精确匹配 | |
| status | `项目状态` 前缀匹配 | 排除含 `进度` 的列 |
| owner | `负责人`、`CD负责人`、`硬装负责人`、`VM负责人`、`软装负责人` | 拆分负责人会汇总为项目负责人 |
| cdOwner | `CD负责人`、`硬装负责人` | SQLite `projects.cd_owner_text` 单独落列 |
| vmOwner | `VM负责人`、`软装负责人` | SQLite `projects.vm_owner_text` 单独落列 |
| progress | `硬装项目进度` 或精确 `进度` | 不匹配 `软装项目进度` |
| startDate | `启动时间`、`启动日期` 精确匹配 | 管理边界字段，不匹配 `平面开始时间` |
| dueDate | `计划开业时间` 等前缀/精确匹配 | 用于展示、筛选和开业边界统计，不作为设计阶段节点 |
| riskLevel | `风险等级` 前缀/精确匹配 | |
| riskNotes | `风险说明`、`风险备注` | |
| updatedAt | `更新时间`、`最后更新时间` | |

负责人责任身份切分只发生在本地责任身份解析、指标过滤和前端展示层。钉钉原始 `负责人`、`CD负责人`、`VM负责人` 字段必须保留原样；完整规则见 [人员与责任身份契约](./personnel-and-responsibility-routing.md)。

## 字段清洗规则

- 字符串会去除前后空格。
- 对象字段优先取 `name`、`text`、`title`、`label`、`value`。
- 数组字段会合并为可读文本。
- 10 位或 13 位数字日期会转换为日期。
- 缺失字段使用“未填写”“未分配”“未分类”等默认值。

## 阶段进度规则

真实钉钉表的项目进度可能是阶段枚举，而不是百分比。系统允许将阶段枚举映射为本地展示进度。

| 阶段关键词 | 本地进度 |
| --- | --- |
| 未开始 / 未安排 | 0 |
| 暂停 | 10 |
| 平面躺平 | 20 |
| 完成复尺 | 25 |
| 完成上会 | 35 |
| 软装方案中 | 45 |
| 施工图 | 55 |
| 待采购 | 65 |
| 施工图完成审核 | 75 |
| 点位已完成 | 80 |
| 施工整改 | 85 |
| 摆场 | 90 |
| 闭环 / 已完成 | 100 |

这只是本地展示和平均进度估算，不写回钉钉。后续如果本地人工维护了最终进度，则本地进度优先于钉钉阶段枚举。

项目明细的“下一提醒”不按 `启动时间` / `计划开业时间` 推导。当前业务阶段按以下责任口径理解：

| 业务段 | 关键字段 | 说明 |
| --- | --- | --- |
| 管理边界 | `启动时间`、`计划开业时间` | 只展示和统计，不提醒设计师“启动 / 开业” |
| 前置协同 | `上会日期`、`复尺时间` | 可提醒补填或催促，不代表硬装开始 |
| 硬装平面图 | `平面开始时间` -> `躺平内部审核结束时间` | 填了平面开始即进入硬装设计责任；填写躺平内部审核结束时间即视为硬装设计师方案交付完成 |
| 硬装施工图 | `施工图初稿完成时间` -> `施工图完成审核时间` | 施工图通常外包，只作为公司项目进度记录，不划入本公司设计师责任制完成口径 |
| 软装点位设计 | `点位完成情况`、`点位完成时间` | 归软装负责人、组长、点位设计师跟进 |
| 软装方案设计 | `软装方案开始时间`、`软装完成情况`、`软装完成时间` | 归软装负责人、组长、软装设计师跟进 |
| 产品清单记录 | `产品清单发出时间` | 业务上理解为外部产品清单到达/接收记录，不推动首页主流程阶段 |
| 后置协同 | `采购时间`、`采购完成情况`、`施工完成时间`、`摆场开始时间`、`摆场时间`、`现场摆场时间`、`摆场文件发出时间(项目群）` | 采购推进和摆场交付保留为公司项目阶段；摆场开始字段表示进入摆场中，摆场文件发出表示摆场结束 |

软装责任制按岗位独立闭环：点位设计师只按点位完成判断，软装设计师只按软装方案完成判断；公司项目层面的软装设计阶段完成才要求两段都完成。

## 字段变更处理

当钉钉表字段变化时：

1. 先只读取字段名和字段结构摘要，不输出业务明细。
2. 若只是括号备注变化，通常无需改配置，重新同步即可。
3. 若 baseName 本身变化，更新 `fieldResolver.mjs` 规则或 `.env` 覆盖。
4. 必要时更新 `projectData.mjs` 的清洗规则。
5. 补充测试。
6. 重新执行同步导入，并检查本地覆盖字段是否产生来源差异。

查看最近一次同步的 `fieldMapping` / `fieldMappingWarnings`（JSON 缓存或 SQLite `field_source_bindings`）可快速定位映射失败原因。
